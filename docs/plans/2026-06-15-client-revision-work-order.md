## 1. 전체 디자인 개선

### REQ-001. 주요 대제목 시인성 강화

대상 화면 또는 메뉴:

- 운영현황
- 콜원장
- 월마감
- 대시보드
- 마스터설정
- 감사로그
- 그 외 같은 수준의 대제목

작업 내용:

- 현재 대제목이 글씨 굵기 정도로만 구분되어 있어 직관성이 부족하다.
- 각 대제목에 배경 색상, 구분 영역, 강조 스타일 등 시각적으로 더 잘 보이는 방식을 적용한다.
- 화면 전체 디자인과 어울리되, 직원이 메뉴나 영역을 빠르게 구분할 수 있어야 한다.

검수 기준:

- 대제목이 본문, 카드, 표 제목과 확실히 구분된다.
- 사용자가 화면을 처음 봤을 때 현재 보고 있는 영역을 즉시 알 수 있다.

## 2. 운영현황

### REQ-002. 객실 표시 순서를 층별 구조로 변경

대상 화면:

- 첫 화면
- 객실현황
- TV 현황판

작업 내용:

- 현재 객실 번호 순서가 섞여 있어 직원이 층별로 보기 불편하다.
- 객실은 위에서 아래로 높은 층에서 낮은 층 순서로 표시한다.
- 객실 배치는 아래 구조를 기준으로 한다.

```text
401 / 402

301 / 302 / 303

201 / 202 / 203

101 / 102 / 103
```

검수 기준:

- 첫 화면, 객실현황, TV 현황판의 객실 정렬이 모두 동일하다.
- 위에서 아래로 4층, 3층, 2층, 1층 순서로 보인다.
- 같은 층 객실은 왼쪽에서 오른쪽으로 객실 번호 오름차순으로 보인다.

### REQ-003. 종료 10분 전 "종료임박" 상태 추가

대상 화면:

- 첫 화면 실시간현황
- 객실현황
- TV 현황판

작업 내용:

- 종료 10분 전부터 카운터 직원이 쉽게 인지할 수 있도록 `종료임박` 상태를 추가한다.
- 상태 흐름은 아래 순서를 기준으로 한다.

```text
사용중 -> 종료임박 -> 종료확인
```

- `종료임박` 상태는 깜빡임, 색상 변경, 강조 배지 등으로 눈에 확실히 띄게 표시한다.
- 세 대상 화면에서 같은 기준과 같은 의미로 표시되어야 한다.

검수 기준:

- 종료 예정 시간이 10분 이하로 남은 객실은 자동으로 `종료임박`으로 표시된다.
- 종료 10분 초과 객실은 기존 `사용중` 상태를 유지한다.
- 종료 이후 확인 단계로 넘어갈 수 있고, `종료확인` 상태와 혼동되지 않는다.
- 첫 화면 실시간현황, 객실현황, TV 현황판에서 모두 동일하게 확인된다.

## 3. 콜원장 및 예약 원장

### REQ-004. 콜/예약 입력 원장에서 정산 관련 정보 숨김

대상 화면:

- 콜/예약 입력 원장
- 카운터 직원 또는 일반 직원이 접근하는 콜원장 화면

작업 내용:

- 콜/예약 입력 원장에는 정보 입력과 결제 관련 정보만 표시한다.
- 일반 직원에게는 아래 항목까지만 표시한다.

표시 유지:

- 결제금액
- 할인

숨김 또는 운영자 전용 화면으로 분리:

- 마사지사1 수당
- 마사지사2 수당
- 귀케어 수당
- 기타 정산 관련 정보

배경:

- 이전 엑셀에서도 마사지사 수당은 숨김 처리되어 있었다.
- 마사지사마다 수당이 다르므로 카운터 직원이나 다른 직원이 해당 내용을 볼 필요가 없다.

검수 기준:

- 카운터 직원 또는 일반 직원 화면에서 마사지사 수당, 귀케어 수당, 기타 정산 정보가 보이지 않는다.
- 운영자 권한 화면에서는 필요한 정산 관련 정보를 확인할 수 있다.
- 정보 숨김으로 인해 예약 등록, 결제 입력, 상태 변경 기능이 깨지지 않는다.

### REQ-005. 할인 적용 시 결제금액 표시 방식 개선

대상 화면:

- 콜/예약 입력 원장
- 결제금액과 할인이 함께 보이는 영역

작업 내용:

- 현재는 할인을 선택하면 결제금액이 자동으로 할인된 금액만 표시되어 혼동될 수 있다.
- 할인 적용 시 결제금액 칸은 아래 형태로 표시한다.

```text
정상가(취소선) + 실제 결제금액
```

- 이전에 제안된 방향을 따르되, 시인성을 더 높인다.

검수 기준:

- 할인이 없는 경우 기존처럼 결제금액을 명확히 표시한다.
- 할인이 있는 경우 정상가와 실제 결제금액을 동시에 확인할 수 있다.
- 정상가는 취소선 등으로 할인 전 금액임을 알 수 있다.
- 실제 결제금액이 더 눈에 잘 들어온다.

### REQ-006. 객실 선택 없이 예약 등록 가능

대상 화면:

- 예약 등록 화면
- 콜/예약 입력 원장
- 예약 그리드

작업 내용:

- 현재는 객실정보를 입력해야 예약 그리드에 추가되는 것으로 보인다.
- 예약 등록 시 객실 선택을 필수값으로 두지 않는다.
- 객실정보 없이도 예약 등록이 가능해야 한다.
- 예약 시점에는 객실을 비워둘 수 있어야 한다.
- 고객이 실제 방문하거나 객실을 배정하는 시점에 객실을 선택할 수 있어야 한다.

배경:

- 예약 후 4~5시간 뒤 방문하는 고객이 있다.
- 워킹 손님도 있어 예약 시점에 객실을 확정하기 어렵다.

검수 기준:

- 객실을 비운 상태로 예약을 저장할 수 있다.
- 객실이 없는 예약도 예약 그리드에 표시된다.
- 이후 실제 배정 시 객실을 선택하고 저장할 수 있다.
- 객실 미선택 예약 때문에 객실현황 또는 매출 집계가 오류를 내지 않는다.

### REQ-007. 선결제 시스템 반영

대상 화면 또는 기능:

- 콜/예약 입력 원장
- 결제수단 선택
- 상태 변경
- 매출 및 결제정보 집계
- 대시보드와 요약 정보

작업 내용:

- 매장은 선결제 시스템으로 운영될 예정이다.
- 현재는 `방문완료` 상태가 되어야 결제금액이 반영되는 것으로 보인다.
- 실제 운영에서는 `사용중` 상태가 되기 전 또는 결제수단을 선택하는 시점에 결제가 완료되는 경우가 대부분이다.
- 결제수단을 선택한 시점부터 매출 및 결제정보가 반영되도록 한다.
- 또는 `사용중` 선택 시점부터 매출 및 결제정보가 반영되도록 한다.
- `방문완료`가 되어야만 매출에 잡히는 구조는 수정한다.

검수 기준:

- 결제수단 선택 직후 해당 건의 결제금액이 매출 및 결제수단별 집계에 반영된다.
- 결제수단 선택 없이 `사용중`으로 변경하는 흐름이 있다면, 그 시점부터 매출 반영 기준이 적용된다.
- `방문완료` 상태 변경 전에도 선결제된 건은 매출에 포함된다.
- 취소, 노쇼, 환불 같은 예외 상태가 있다면 기존 정책에 맞게 매출 반영 여부가 일관되게 처리된다.

확인 필요:

- 결제수단 선택과 `사용중` 선택이 모두 발생하는 경우, 중복 집계되지 않도록 단일 기준이 필요하다.
- 취소 또는 환불 시 이미 반영된 선결제 매출을 차감할지 정책 확인이 필요하다.

### REQ-008. 상단 요약 정보 유지 및 결제수단별 집계 추가

대상 화면:

- 콜원장 상단 요약 정보
- 예약 원장 상단 요약 정보

작업 내용:

- 아래 상태 요약은 유지한다.

유지할 상태 요약:

- 예약건수
- 사용중
- 청소중
- 방문완료
- 노쇼
- 취소

- 아래 금액 요약도 현재처럼 표시한다.

유지할 금액 요약:

- 결제합계
- 할인합계
- 지출합계

- 추가로 결제수단별 집계를 작게라도 표시한다.

추가할 결제수단별 집계 예시:

```text
현금
15,000,000

카드
20,000,000

계좌
10,000,000

기타
5,000,000
```

운영 목적:

- 카운터 직원이 오늘 현금으로 얼마 받았는지 확인할 수 있어야 한다.
- 지출이 얼마 나갔는지 확인할 수 있어야 한다.
- 현재 돈통에 얼마가 남아있는지 쉽게 확인할 수 있어야 한다.
- 현금결제가 90% 이상일 것으로 예상되지만, 카드, 계좌, 기타 결제도 확인 가능해야 한다.

검수 기준:

- 상단 요약에서 기존 상태 카운트와 금액 합계를 계속 확인할 수 있다.
- 결제수단별 금액이 현금, 카드, 계좌, 기타로 나뉘어 보인다.
- 선결제 반영 기준과 결제수단별 집계 기준이 일치한다.
- 화면이 좁아져도 결제수단별 집계가 읽기 어렵게 깨지지 않는다.

확인 필요:

- "돈통에 남아있는 금액"을 별도 값으로 표시할지, 현금 결제합계와 지출합계만 보여줄지 결정이 필요하다.
- 별도 표시한다면 기본 계산식은 `현금 결제합계 - 현금 지출합계`인지 확인이 필요하다.

### REQ-009. 예약건수 집계 기준 수정

대상 화면:

- 콜원장 상단 요약
- 예약 원장 상단 요약
- 예약건수를 표시하는 모든 요약 영역

작업 내용:

- 현재 예약건수는 `예약` 상태일 때만 카운트되고, `방문완료` 등으로 변경되면 0건으로 바뀌는 것으로 보인다.
- 예약건수는 상태와 관계없이 원장 그리드에 등록된 전체 건수 기준으로 집계한다.
- 의뢰자가 제시한 기준은 아래와 같다.

```text
총 예약 50건

방문완료 40건
노쇼 5건
취소 5건

-> 예약건수 50건
```

검수 기준:

- 예약건수는 예약 상태만 세지 않고, 그리드에 등록된 전체 예약 등록 건수를 의미한다.
- 방문완료, 노쇼, 취소 상태로 바뀌어도 예약건수 총합에서 빠지지 않는다.
- 상태별 카운트 합계와 예약건수 총합이 논리적으로 맞는다.

- `지급완료`는 정산금 지급 여부를 명확히 확인하고 관리할 수 있어야 한다.

검수 기준:

- 일일정산 화면에서 각 마사지사 또는 정산 대상별 지급완료 여부를 확인할 수 있다.

- 필요 시 지급완료 상태를 운영자가 변경할 수 있다.

확인 필요:

- `지급완료`를 단순 체크박스로 둘지, 지급일시와 처리자를 함께 기록할지 결정이 필요하다.

## 5. 오늘 대시보드

### REQ-011. 오늘 대시보드 순이익 계산 기준 수정

대상 화면:

- 오늘 대시보드

현재 구성:

- 결제합계
- 순매출
- 할인합계
- 지출합계
- 귀케어
- 마사지사정산

작업 내용:

- 순이익은 실제 운영상 최종적으로 남는 금액 기준으로 계산한다.
- 당일 지급 비용인 `일일인센`을 오늘 대시보드에 추가한다.
- 아래 항목을 비용 또는 차감 항목으로 명확히 표시한다.

오늘 대시보드 표시 항목 예시:

```text
결제합계
10,000,000

할인합계
1,000,000

일일인센 합계
1,000,000 (빨간색 표시)

지출합계
500,000 (빨간색 표시)

마사지사 정산
2,000,000 (빨간색 표시)

귀케어 정산
500,000 (빨간색 표시)

순이익
6,000,000
```

계산 기준:

- `결제합계`가 실제 결제금액 합계라면, 순이익은 아래 기준을 따른다.

```text
순이익 = 결제합계 - 일일인센 합계 - 지출합계 - 마사지사 정산 - 귀케어 정산
```

- 이 기준에서는 `할인합계`가 이미 실제 결제금액에 반영되어 있으므로 순이익에서 다시 차감하지 않는다.

검수 기준:

- 오늘 대시보드에 일일인센 합계가 표시된다.
- 일일인센 합계, 지출합계, 마사지사 정산, 귀케어 정산은 비용으로 인지되도록 빨간색 등으로 표시된다.
- 순이익이 실제 남는 금액 기준으로 계산된다.
- 결제합계와 할인합계의 의미가 중복 차감되지 않게 일관된다.

확인 필요:

- 원문 예시의 숫자는 `10,000,000 - 1,000,000 - 500,000 - 2,000,000 - 500,000 = 6,000,000` 구조로 보이며, 할인합계는 표시만 하고 순이익에서 다시 빼지 않는 전제로 해석된다.
- 기존 시스템의 `결제합계`가 할인 전 정상가 합계라면 계산식이 달라지므로 데이터 정의 확인이 필요하다.

### REQ-012. 결제합계와 순이익 강조 표시

대상 화면:

- 오늘 대시보드

작업 내용:

- 결제합계와 순이익은 가장 중요한 지표이다.
- 다른 항목보다 더 눈에 띄게 표시한다.
- 강조 방식은 카드 크기, 색상, 강조 효과 등을 사용할 수 있다.

검수 기준:

- 사용자가 대시보드를 봤을 때 결제합계와 순이익이 가장 먼저 눈에 들어온다.
- 강조가 과하지 않고 다른 비용 항목과 시각적으로 구분된다.

## 6. 월간 대시보드

### REQ-013. 월간 대시보드도 오늘 대시보드와 같은 순이익 개념 적용

대상 화면:

- 월간 대시보드

작업 내용:

- 월간 대시보드도 오늘 대시보드와 동일한 개념으로 계산하고 표시한다.
- 월간 순이익은 월간 비용까지 반영한 최종 순이익이어야 한다.
- 월간 순이익에는 아래 월간 비용이 반영되어야 한다.

월간 비용:

- 운영팀 월인센
- 마사지사 만근수당
- 마사지사 갯수왕

검수 기준:

- 월간 대시보드의 순이익은 일일 비용과 월간 비용을 모두 반영한 최종 금액으로 표시된다.
- 운영팀 월인센, 마사지사 만근수당, 마사지사 갯수왕이 월간 순이익 계산에 포함된다.
- 오늘 대시보드와 월간 대시보드의 계산 개념이 서로 어긋나지 않는다.

### REQ-014. 일일비용과 월간비용 구분 적용

대상 화면:

- 월간 대시보드
- 월마감 또는 월간 정산 관련 화면

작업 내용:

- 일일비용과 월간비용을 구분하여 적용한다.

일일비용:

- 마사지사정산
- 귀케어 정산
- 운영팀 일일인센

월간비용:

- 운영팀 월인센
- 마사지사 만근수당
- 마사지사 갯수왕

검수 기준:

- 월간 대시보드에서 일일비용과 월간비용이 구분되어 보이거나, 적어도 계산 기준이 명확히 분리된다.
- 월간 순이익 계산 시 일일비용 누계와 월간비용이 모두 반영된다.
- 같은 비용이 중복 차감되지 않는다.

## 8. 전체 검수 체크리스트

- [ ] 주요 대제목의 시인성이 개선되었다.
- [ ] 첫 화면, 객실현황, TV 현황판의 객실이 층별 구조로 정렬된다.
- [ ] 종료 10분 전 `종료임박` 상태가 표시된다.
- [ ] `종료임박` 상태가 세 대상 화면에서 모두 눈에 띄게 표시된다.
- [ ] 일반 직원용 콜원장에 마사지사 수당과 정산 관련 정보가 보이지 않는다.
- [ ] 운영자 화면에서 정산 관련 정보를 확인할 수 있다.
- [ ] 할인 적용 시 정상가 취소선과 실제 결제금액이 함께 보인다.
- [ ] 객실 없이 예약을 등록할 수 있다.
- [ ] 객실 없는 예약을 이후 실제 배정 시점에 객실과 연결할 수 있다.
- [ ] 결제수단 선택 또는 `사용중` 선택 시점부터 매출 및 결제정보가 반영된다.
- [ ] `방문완료` 전 선결제 건도 집계에 반영된다.
- [ ] 상단 요약에 기존 상태 카운트가 유지된다.
- [ ] 상단 요약에 결제합계, 할인합계, 지출합계가 유지된다.
- [ ] 상단 요약에 현금, 카드, 계좌, 기타 결제수단별 집계가 추가된다.
- [ ] 예약건수는 상태와 관계없이 원장 그리드 전체 등록 건수로 계산된다.
- [ ] 일일정산 화면에 `지급완료` 항목이 추가된다.
- [ ] `지급완료` 상태가 저장되고 확인 가능하다.
- [ ] 오늘 대시보드에 일일인센 합계가 추가된다.
- [ ] 오늘 대시보드 순이익이 실제 남는 금액 기준으로 계산된다.
- [ ] 오늘 대시보드의 비용성 항목이 빨간색 등으로 구분된다.
- [ ] 오늘 대시보드에서 결제합계와 순이익이 강조된다.
- [ ] 월간 대시보드에 오늘 대시보드와 같은 순이익 개념이 적용된다.
- [ ] 월간 순이익에 운영팀 월인센, 마사지사 만근수당, 마사지사 갯수왕이 반영된다.
- [ ] 월간 대시보드에서 일일비용과 월간비용이 구분되어 적용된다.

## 9. 구현 전 확인 필요 사항

1. 선결제 매출 반영 기준을 `결제수단 선택 시점`으로 할지, `사용중 선택 시점`으로 할지 최종 확정이 필요하다.
2. 선결제 후 취소, 노쇼, 환불이 발생하는 경우 매출과 결제수단별 집계를 어떻게 차감할지 정책 확인이 필요하다.
3. 돈통 잔액을 별도 표시할 경우 계산식을 `현금 결제합계 - 현금 지출합계`로 볼지 확인이 필요하다.
4. `지급완료` 항목을 단순 체크 상태로 둘지, 지급일시와 처리자까지 저장할지 확인이 필요하다.
5. `결제합계`가 할인 후 실제 결제금액인지, 할인 전 정상가 합계인지 확인이 필요하다. 원문 예시는 할인합계를 순이익에서 다시 차감하지 않는 전제로 보인다.

## 10. 일정 및 커뮤니케이션 참고

- 의뢰자는 전체적인 진행 방향에 만족한다고 전달했다.
- 현재 공사기간이 늘어나 오픈 예정일은 6월 말 또는 7월 초로 예상된다고 전달했다.
- 의뢰자는 천천히 진행해도 괜찮다고 언급했다.
- 단, 위 수정요청사항은 오픈 전 운영 흐름과 직접 연결되므로 선결제, 객실 배정, 매출 집계, 정산 관련 항목은 우선순위를 높게 두는 것이 좋다.
