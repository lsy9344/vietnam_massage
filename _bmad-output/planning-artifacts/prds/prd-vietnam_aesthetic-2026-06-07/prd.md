# PRD: 베트남 에스테틱 ERP

## 0. 문서 목적

이 PRD는 기존 `sheet.xlsx`로 운영 중인 베트남 에스테틱/마사지 매장 업무를 웹 기반 ERP로 전환하기 위한 제품 요구사항 문서다. 대상 독자는 의사결정자, PM, UX 설계자, 아키텍트, 개발자, QA 담당자이며, 이 문서는 기능 보존 범위, 사용자 여정, 기능 요구사항, 비기능 요구사항, 성공 기준을 정의한다.

본 PRD는 원본 엑셀의 12개 시트와 숨김 시트 `목록`, 그리고 기존 분석 문서 `client_erp_specification.md`, `sheet_erp_design.md`, `_bmad-output/planning-artifacts/briefs/brief-vietnam_aesthetic-2026-06-07/brief.md`를 기준으로 한다. 기술스택과 구현 제약은 `addendum.md`에 별도 보존한다.

## 1. 비전

베트남 에스테틱 ERP는 현재 엑셀 한 파일에 들어 있는 예약, 객실 현황, 콜 입력, 매출, 지출, 마사지사 정산, 귀케어 정산, 운영팀 인센, 월마감을 기능 누락 없이 웹 시스템으로 이전하는 제품이다. 이 제품의 1차 목표는 새 기능을 많이 붙이는 것이 아니라, 이미 매장에서 돌아가는 엑셀 업무 규칙을 정확히 보존하고 운영자가 더 빠르고 안전하게 사용할 수 있게 만드는 것이다.

ERP의 중심은 `실시간콜입력`을 대체하는 콜 원장이다. 모든 예약, 사용중, 청소중, 방문완료, 노쇼, 취소 상태는 콜 원장을 기준으로 기록되고, 객실 현황, TV 현황판, 대시보드, 일정산, 월마감은 이 원장을 읽어 계산한다. 엑셀의 셀 좌표와 반복 행 구조는 그대로 복제하지 않지만, 업무상 의미 있는 입력값, 계산값, 상태 표시, 정산 결과는 동일하게 보존한다.

주인은 첫 화면에서 실시간 객실/콜 현황을 한눈에 보고 매장 상태를 즉시 이해할 수 있어야 한다. 대시보드는 화려하지만 복잡하지 않고, 매출·콜·객실·정산·인센의 핵심 흐름이 카드, 그래프, 순위, 상태 색상으로 명확하게 드러나야 한다.

## 2. 대상 사용자

### 2.1 주요 사용자와 해야 할 일

- **주인/관리자** — 실시간 운영 상태, 매출, 객실 가동, 정산 위험, 월마감 결과를 한눈에 보고 설정, 직원, 수당, 마감, 재오픈을 통제한다.
- **카운터** — 예약/콜을 입력하고 상태, 결제수단, 할인구분, 담당자, 확인값, 비고를 빠르게 관리한다.
- **웨이터** — 객실별 상태, 남은 시간, 종료확인, 청소 상태, 안내 문구를 확인한다.
- **정산 담당** — 마사지사, 귀케어, 운영팀의 일일 정산과 월마감을 검토한다.
- **조회 전용 사용자** — TV 현황판, 객실 현황, 오늘 대시보드를 확인한다.

### 2.2 v1 비사용자

- 고객 또는 멤버십 회원은 v1 사용자 범위가 아니다.
- 외부 회계사, 마케팅 담당자, CRM 운영자는 v1 사용자 범위가 아니다.
- 모바일 앱 전용 사용자는 v1 범위가 아니다.

### 2.3 핵심 사용자 여정

- **UJ-1. 주인이 실시간 객실/콜 현황으로 매장 상태를 파악한다.**
  - **페르소나 + 맥락:** noah는 매장 운영자이며, 하루 중 여러 번 현재 객실 상태와 콜 흐름을 확인한다.
  - **진입 상태:** 로그인 후 ERP 첫 화면에 진입한다.
  - **경로:** 실시간 객실/콜 현황 화면에서 11개 객실 카드, 오늘 예약/사용중/청소중/방문완료 수, 결제합계, 코스별 콜 현황, 남은 시간 알림을 본다. 필요한 경우 특정 객실 카드나 콜 목록을 눌러 상세를 확인한다.
  - **가치 순간:** 어떤 객실이 사용중인지, 어떤 객실이 종료확인 상태인지, 오늘 매출과 콜이 어느 정도인지 즉시 파악한다.
  - **해결 상태:** 별도 엑셀 시트를 이동하지 않고 운영 판단을 내린다.

- **UJ-2. 카운터가 예약/콜을 입력하고 방문완료 처리를 한다.**
  - **페르소나 + 맥락:** 카운터 직원은 고객 방문 또는 예약 전화를 받으며 빠르게 콜을 기록해야 한다.
  - **진입 상태:** 로그인 후 콜/예약 입력 화면에 진입한다.
  - **경로:** 날짜와 시간, 객실, 코스, 고객/메모, 마사지사1, 마사지사2, 귀케어 담당, 상태, 할인구분, 결제수단, 비고, 확인값을 입력한다. 상태가 `방문완료`가 되면 결제금액, 수당, 귀케어 풀, 콜인정이 자동 계산된다.
  - **가치 순간:** 엑셀 수식처럼 계산 결과가 즉시 반영되지만, 누가 어떤 값을 바꿨는지 이력이 남는다.
  - **해결 상태:** 대시보드, 객실 현황, 정산 화면이 같은 콜 원장을 즉시 반영한다.

- **UJ-3. 웨이터가 객실 상태와 안내 문구를 확인한다.**
  - **페르소나 + 맥락:** 웨이터는 객실 청소, 입실 가능 여부, 종료확인을 빠르게 판단해야 한다.
  - **진입 상태:** 객실 현황 또는 TV 현황판을 조회한다.
  - **경로:** 객실별 상태 색상, 코스, 담당, 시작시간, 남은분, 종료예정, 안내 문구를 확인한다.
  - **가치 순간:** `사용중`, `청소중`, `예약`, `종료확인`, `빈방` 상태가 색상과 문구로 명확히 드러난다.
  - **해결 상태:** 웨이터는 원장 수정 없이 필요한 행동을 결정한다.

- **UJ-4. 정산 담당이 일일 정산과 월마감을 검토한다.**
  - **페르소나 + 맥락:** 정산 담당자는 마사지사, 귀케어, 운영팀 지급액을 정확하게 확인해야 한다.
  - **진입 상태:** 정산 화면에서 운영월과 날짜를 선택한다.
  - **경로:** 방문완료 콜 기준 마사지사 수당, 귀케어 N분의1 지급, 운영팀 인센, 출퇴근/대기시간/만근 인정, 월 총콜과 최종지급액을 검토한다.
  - **가치 순간:** 원본 엑셀과 같은 지급 계산이 모듈별로 설명 가능하게 나타난다.
  - **해결 상태:** 월마감 미리보기, 검토, 확정, 잠금으로 지급액을 고정한다.

## 3. 용어

- **운영월** — 기준월 단위의 운영 기간. 날짜 목록, 집계, 정산, 월마감의 기준이다.
- **콜 원장** — `실시간콜입력`을 대체하는 핵심 거래 원장. 예약, 방문, 객실, 코스, 담당자, 상태, 결제, 할인, 정산 기초값을 보유한다.
- **서비스 콜** — 한 건의 예약 또는 방문 기록. 상태가 `방문완료`일 때만 매출, 수당, 콜인정에 반영된다.
- **방문완료** — 매출, 마사지사 수당, 귀케어 풀, 콜인정이 발생하는 유일한 완료 상태.
- **객실 현황** — 객실별 최신 `사용중`, `청소중`, `예약` 콜을 기준으로 현재 상태, 남은 시간, 안내 문구를 보여주는 조회 화면.
- **TV 현황판** — 카운터 또는 매장 TV에 표시하는 조회 전용 객실 카드 화면.
- **마사지사 수당** — 방문완료 콜의 코스와 담당 마사지사별 수당표에 따라 계산되는 금액.
- **귀케어 풀** — B/E 코스 등 귀케어 포함 코스에서 발생하는 일별 분배 대상 금액.
- **운영팀 인센** — 일일 총콜 및 월 총콜 구간에 따라 운영팀에게 지급되는 인센티브.
- **월마감** — 운영월의 매출, 지출, 수당, 인센, 보너스, 최종지급액을 확정하고 잠그는 절차.
- **마감 스냅샷** — 월마감 확정 시점의 원장, 정책, 지급 계산 근거를 보존한 기록.
- **감사 로그** — 지급액이나 운영 상태에 영향을 주는 변경의 행위자, 변경 전후값, 시각을 보존하는 불변 이력.

## 4. 기능 요구사항

### 4.1 실시간 객실/콜 현황 첫 화면

**설명:** 로그인 후 첫 화면은 실시간 객실/콜 현황이다. 이 화면은 원본 `웨이터리스트`, `TV현황판`, `오늘대시보드`의 핵심 정보를 통합해 주인이 한눈에 이해할 수 있도록 보여준다. 시각 스타일은 화려하고 선명하지만, 정보 구조는 단순해야 한다.

#### FR-1: 첫 화면 실시간 현황

시스템은 로그인 후 사용자를 실시간 객실/콜 현황 화면으로 안내한다. 실현: UJ-1.

**검증 조건:**
- 첫 화면에서 11개 객실의 현재 상태가 보인다.
- 첫 화면에서 오늘 예약, 사용중, 청소중, 방문완료, 노쇼, 취소 요약이 보인다.
- 첫 화면에서 오늘 결제합계, 순매출, 코스별 방문완료 수가 보인다.

#### FR-2: 객실 카드 시각화

시스템은 각 객실을 카드 형태로 표시하고 상태별 색상, 코스, 담당자, 시작시간, 남은분, 종료예정을 표시한다. 실현: UJ-1, UJ-3.

**검증 조건:**
- 상태값 `사용중`, `청소중`, `예약`, `종료확인`, `빈방`마다 구분되는 색상 토큰이 적용된다.
- `사용중`이고 남은분이 0이면 표시 상태가 `종료확인`으로 바뀐다.
- TV 현황판과 객실 현황은 같은 상태 계산 결과를 사용한다.

#### FR-3: 주인용 그래프 대시보드

시스템은 주인용 화면에 일별 매출 추이, 코스별 콜 비중, 마사지사 콜/정산 순위, 객실 상태 분포, 노쇼/취소 추이, 운영팀 인센 달성률을 시각화한다. 실현: UJ-1.

**검증 조건:**
- 그래프는 원본 엑셀의 오늘/월간 KPI와 콜 원장 집계에서 계산된다.
- 그래프는 `방문완료` 기준 매출과 콜만 매출/정산 지표에 반영한다.
- 그래프는 복잡한 분석 필터보다 날짜/운영월 선택과 핵심 KPI 이해를 우선한다.

### 4.2 마스터 설정

**설명:** 원본 `직원DB`, `TV설정`, `설정_코스수당`, 숨김 시트 `목록`의 설정 기능을 ERP 마스터로 이전한다. 엑셀의 표시명은 보존하되, ERP 내부에서는 이름 문자열이 아니라 고유 ID를 기준으로 참조한다.

#### FR-4: 운영월 관리

관리자는 운영월을 생성하고 상태를 작성중, 검토중, 마감확정, 잠금으로 관리할 수 있다.

**검증 조건:**
- 운영월은 `YYYY-MM` 기준으로 월 시작일과 월 종료일을 가진다.
- 운영월 상태는 월마감 상태와 연결된다.
- 날짜 목록과 월간 집계는 운영월 날짜 조건으로 계산된다.

#### FR-5: 객실 마스터

관리자는 11개 객실을 관리할 수 있으며, 표준 표시명은 `101 호실` 형식을 사용한다.

**검증 조건:**
- 기본 객실은 `101 호실`, `102 호실`, `103 호실`, `201 호실`, `202 호실`, `203 호실`, `301 호실`, `302 호실`, `303 호실`, `401 호실`, `402 호실`이다.
- 객실은 고유 ID와 표시명을 분리한다.
- 숨김 목록의 `1번방` 형식은 이관 참조값으로만 보존하고 운영 표준 표시명으로 쓰지 않는다.

#### FR-6: 직원 마스터

관리자는 운영팀 5명, 귀케어팀 4명, 마사지사 50명의 직원 정보를 관리할 수 있다.

**검증 조건:**
- 직원은 고유 ID, 이름, 역할, 주/야간, 기본급, 연락처, 생일, 입사일, 재직상태를 가진다.
- 이름 변경은 과거 정산 스냅샷을 변경하지 않는다.
- 삭제 대신 비활성 처리를 지원한다.

#### FR-7: 코스 마스터

관리자는 A~E 코스의 이름, 시간, 기본판매가, 운영팀 콜인정, 귀케어 풀/콜, 마사지사2 필요 여부, TV 표시명을 관리할 수 있다.

**검증 조건:**
- 기본 코스는 A 60분, B 90분, C 90분, D 90분 2:1, E 120분이다.
- D코스는 `마사지사2 필요`가 Y인 코스로 관리된다.
- 코스 변경은 적용월과 이력을 가진다.

#### FR-8: 수당과 인센 정책 관리

관리자는 마사지사 개인별 코스 수당, 운영팀 일일 인센, 운영팀 월 인센 기준을 관리할 수 있다.

**검증 조건:**
- 마사지사 수당은 마사지사, 코스, 금액, 적용 시작월, 적용 종료월을 가진다.
- 원본 엑셀의 마사지사5~50 수당 0원은 초기 이관값으로 보존하고 관리자 수정이 가능하다.
- 운영팀 일일 인센 30/40/50콜 기준과 월 인센 1000~1500콜 기준을 보존한다.

#### FR-9: 코드와 시간 슬롯 관리

관리자는 상태, 결제수단, 할인구분, 확인값, 근무상태, 시간 슬롯을 관리할 수 있다.

**검증 조건:**
- 상태값은 `예약`, `사용중`, `청소중`, `방문완료`, `노쇼`, `취소`를 포함한다.
- 결제수단은 `현금`, `카드`, `계좌`, `기타`를 포함한다.
- 할인구분은 `일주일내방문`, `생일자`, `후기작성`을 포함한다.
- 입력 시간 슬롯은 실제 `실시간콜입력` 드롭다운 기준인 `11:00`부터 `01:00`까지 30분 단위로 제공한다.

### 4.3 예약/콜 원장

**설명:** 원본 `실시간콜입력`의 A:S 입력/계산 구조와 U:X 일별 지출/요약 구조를 ERP의 핵심 원장으로 이전한다.

#### FR-10: 서비스 콜 입력

카운터는 날짜, 시간, 객실, 코스, 고객/메모, 마사지사1, 마사지사2, 귀케어 담당, 상태, 할인구분, 결제수단, 비고, 확인값을 입력할 수 있다. 실현: UJ-2.

**검증 조건:**
- 일자별 100개 슬롯에 해당하는 입력량을 처리할 수 있다.
- 월 최대 31일의 콜 원장을 운영월 기준으로 조회할 수 있다.
- 입력 필드는 원본 `실시간콜입력` A:S의 업무 의미를 모두 보존한다.

#### FR-11: 서비스 콜 상태 관리

카운터는 서비스 콜 상태를 `예약`, `사용중`, `청소중`, `방문완료`, `노쇼`, `취소`로 변경할 수 있다.

**검증 조건:**
- 상태 변경 이력이 저장된다.
- `방문완료`가 아닌 상태는 매출, 수당, 귀케어 풀, 콜인정에 포함되지 않는다.
- 상태 변경은 감사 로그 대상이다.

#### FR-12: 방문완료 계산

시스템은 상태가 `방문완료`일 때만 결제금액, 마사지사1수당, 마사지사2수당, 귀케어 풀, 콜인정을 계산한다.

**검증 조건:**
- `방문완료`가 아니면 결제금액, 수당, 귀케어 풀, 콜인정은 0이다.
- `방문완료`이면 결제금액은 코스 기본판매가에서 할인금액을 차감한 값이다.
- 콜인정은 `방문완료`일 때 1이다.

#### FR-13: 할인 계산

시스템은 할인구분 값이 있으면 원본 엑셀처럼 고정 100,000원을 할인한다.

**검증 조건:**
- 할인구분이 비어 있으면 할인금액은 0이다.
- 할인구분이 `일주일내방문`, `생일자`, `후기작성` 중 하나이면 할인금액은 100,000이다.
- 할인정책 다양화는 v1 범위가 아니다.

#### FR-14: D코스 2인 검증

시스템은 D코스 서비스 콜에서 마사지사2 입력을 필수로 검증한다.

**검증 조건:**
- D코스에서 마사지사2가 없으면 저장 또는 방문완료 처리를 막고 오류를 표시한다.
- A/B/C/E 코스는 마사지사2가 선택 사항이다.
- 마사지사1과 마사지사2가 모두 있는 경우 각 담당자 수당을 별도로 계산한다.

#### FR-15: 일별 지출과 콜 요약

카운터 또는 관리자 권한 사용자는 일자별 지출금액, 내용, 담당자, 비고를 입력할 수 있으며, 시스템은 원본 `실시간콜입력` U:X 영역의 일별 요약을 계산해 보여준다.

**검증 조건:**
- 지출은 일자와 운영월에 연결된다.
- 일별 지출합계와 순매출 계산에 반영된다.
- 지출 수정은 감사 로그 대상이다.
- 일별 요약은 예약건수, 방문완료, 노쇼/취소, 결제합계, 마사지사정산, 귀케어풀, 할인합계, 지출합계, 순매출을 포함한다.
- 코스별 요약은 A~E 코스별 방문완료 수, 할인건수, 마사지사 담당 수를 포함한다.

### 4.4 객실 현황과 TV 현황판

**설명:** 원본 `웨이터리스트`와 `TV현황판`을 ERP 조회 화면으로 이전한다. 두 화면은 같은 객실 상태 계산을 사용하며, TV 현황판은 입력 없는 전체화면 조회 전용으로 제공한다.

#### FR-16: 객실 상태 계산

시스템은 객실별 최신 `사용중`, `청소중`, `예약` 서비스 콜을 찾아 객실 상태를 계산한다. 실현: UJ-3.

**검증 조건:**
- 활성 콜이 없으면 `빈방`으로 표시한다.
- 최신 활성 콜은 객실, 상태, 시작시간 기준으로 결정된다.
- 상태 계산은 정산 또는 월마감 데이터를 변경하지 않는다.
- 객실 표시 상태는 서비스 콜 상태 전환을 통해 바뀌며, 별도의 정산 데이터 변경으로 처리하지 않는다.

#### FR-17: 남은 시간과 종료확인

시스템은 코스 시간과 시작시간을 기준으로 남은분과 종료예정을 계산한다.

**검증 조건:**
- 시작시간이 자정을 넘는 운영 흐름에서도 종료예정이 올바르게 계산된다.
- `사용중` 상태에서 남은분이 0이면 `종료확인`으로 표시한다.
- 남은분은 음수로 표시하지 않는다.

#### FR-18: 웨이터 안내 문구

시스템은 객실 상태에 맞는 웨이터 안내 문구를 표시한다.

**검증 조건:**
- `사용중`, `청소중`, `예약`, `종료확인`, `빈방`별 안내 문구가 구분된다.
- 안내 문구는 설정 또는 코드로 관리할 수 있다.
- TV 현황판에는 필요한 핵심 문구만 노출한다.

#### FR-19: TV 현황판

시스템은 11개 객실을 대형 카드 형태로 보여주는 조회 전용 TV 현황판을 제공한다.

**검증 조건:**
- TV 현황판은 입력 기능을 제공하지 않는다.
- 자동 새로고침 또는 실시간 갱신을 지원한다.
- 상태 색상과 표시값은 객실 현황 화면과 일치한다.

### 4.5 일정산

**설명:** 원본 `마사지사일정산`, `귀케어일정산`, `운영팀근무인센`의 일별 계산 기능을 ERP로 이전한다.

#### FR-20: 마사지사 출퇴근 입력

정산 담당자는 마사지사별 출근시간과 퇴근시간을 입력할 수 있다. 실현: UJ-4.

**검증 조건:**
- 마사지사 50명의 일별 출퇴근 입력을 지원한다.
- 퇴근시간이 출근시간보다 빠르면 자정 이후 퇴근으로 계산한다.
- 대기시간 8시간 이상이면 만근으로 인정한다.

#### FR-21: 마사지사 일일정산

시스템은 마사지사가 마사지사1 또는 마사지사2 어느 칸에 배정되어도 담당 콜로 인정하고 일일정산을 계산한다.

**검증 조건:**
- 코스별 담당 콜 수는 `방문완료` 콜만 집계한다.
- 당일정산은 해당 마사지사에게 배정된 수당 합계다.
- 마사지사2 담당 콜도 총콜과 정산에 포함된다.

#### FR-22: 귀케어 근무상태 입력

정산 담당자는 귀케어사 4명의 일별 근무상태를 입력할 수 있다.

**검증 조건:**
- 근무상태는 `정상`, `휴무`, `지각`, `조퇴`, `결근`을 지원한다.
- 정상 상태만 귀케어 지급 대상이다.
- 근무상태 변경은 감사 로그 대상이다.

#### FR-23: 귀케어 일일정산

시스템은 방문완료 콜의 귀케어 풀을 해당일 정상 근무 귀케어사에게 균등 분배한다.

**검증 조건:**
- 당일 귀케어 풀은 방문완료 서비스 콜의 귀케어 풀 합계다.
- 정상근무자수가 0명이면 지급액은 0원으로 처리한다.
- 개인 지급액은 정상 근무자에게만 발생한다.

#### FR-24: 운영팀 근무상태와 일일 인센

정산 담당자는 운영팀 5명의 일별 근무상태를 입력하고 시스템은 일일 인센을 계산한다.

**검증 조건:**
- 운영팀 대상자는 팀장, 카운터1, 카운터2, 웨이터1, 웨이터2다.
- 일 총콜 30/40/50 이상이면 개인별 50,000/100,000/200,000을 지급한다.
- 정상 상태인 운영팀 직원만 일일 인센을 받는다.

#### FR-25: 운영팀 월 인센 미리보기

시스템은 운영월 총콜 기준으로 운영팀 월 인센을 계산해 미리 보여준다.

**검증 조건:**
- 월 총콜 1000/1100/1200/1300/1400/1500 이상 구간별 전체 월인센을 적용한다.
- 분배율은 팀장 30%, 카운터팀 35%, 웨이터팀 35%다.
- 카운터팀과 웨이터팀 내부 인원별 분배가 표시된다.

### 4.6 월마감

**설명:** 원본 `월마감` 시트의 월 지급액 계산과 ERP 운영 안전장치를 결합한다.

#### FR-26: 월마감 미리보기

정산 담당자는 운영월의 마사지사, 운영팀, 귀케어 지급액을 미리보기로 확인할 수 있다. 실현: UJ-4.

**검증 조건:**
- 마사지사별 월 총콜, 월 정산액, 만근 인정일, 만근수당, 갯수왕 순위/수당, 최종지급액이 표시된다.
- 운영팀 월 총콜, 월인센 전체, 일일인센 합계가 표시된다.
- 귀케어 지급 합계와 마사지사 최종 지급 합계가 표시된다.

#### FR-27: 만근수당 계산

시스템은 마사지사 만근 인정일이 20일 이상이면 만근수당 2,000,000을 지급한다.

**검증 조건:**
- 만근 인정일은 일별 대기시간 8시간 이상인 날의 합계다.
- 만근 인정일이 19일이면 만근수당은 0이다.
- 만근 인정일이 20일 이상이면 만근수당은 2,000,000이다.

#### FR-28: 갯수왕 수당 계산

시스템은 월 총콜 40콜 이상 마사지사를 대상으로 1~3위 갯수왕 수당을 계산한다.

**검증 조건:**
- 1위는 5,000,000, 2위는 3,000,000, 3위는 1,000,000이다.
- 월 총콜 40콜 미만 마사지사는 순위 수당 대상이 아니다.
- 순위와 수당 산출 근거가 월마감 화면에 표시된다.

#### FR-29: 월마감 확정과 잠금

관리자 또는 정산 담당 권한 사용자는 월마감을 미리보기, 검토, 확정, 잠금 단계로 처리할 수 있다.

**검증 조건:**
- 확정 시 마감 스냅샷이 저장된다.
- 잠금 상태의 운영월은 일반 사용자의 지급 영향 데이터 수정을 막는다.
- 확정과 잠금은 감사 로그 대상이다.

#### FR-30: 월마감 재오픈

관리자는 사유 입력 후 잠긴 월마감을 재오픈할 수 있다.

**검증 조건:**
- 재오픈은 관리자 권한에서만 가능하다.
- 재오픈 사유가 없으면 처리할 수 없다.
- 재오픈 전후 상태와 행위자는 감사 로그에 기록된다.

### 4.7 대시보드와 리포트

**설명:** 원본 `오늘대시보드`를 보존하면서 주인용 그래프/시각화를 강화한다. 그래프는 엑셀에 없던 장식용 기능이 아니라, 원본 KPI를 더 잘 읽기 위한 표현 방식이다.

#### FR-31: 오늘 KPI 대시보드

시스템은 조회날짜 기준 오늘 예약건수, 방문완료 콜, 노쇼, 취소, 결제합계, 마사지사 담당콜, 마사지사 정산, 코스별 방문완료를 보여준다.

**검증 조건:**
- KPI는 조회날짜 변경 시 갱신된다.
- 매출과 정산은 `방문완료` 기준으로 계산된다.
- 원본 `오늘대시보드`의 표시 항목을 모두 포함한다.

#### FR-32: 월간 KPI 대시보드

시스템은 운영월 기준 월 방문완료 콜, 예약건수, 노쇼, 취소, 방문완료 매출을 보여준다.

**검증 조건:**
- 월간 집계는 엑셀 행 범위가 아니라 운영월 날짜 조건으로 계산된다.
- 월마감 확정 후에는 확정 스냅샷 기준 값을 조회할 수 있다.
- 월간 KPI와 오늘 KPI의 계산 기준이 충돌하지 않는다.

#### FR-33: 주인용 시각화 리포트

시스템은 일별 매출 추이, 코스별 콜/매출 비중, 마사지사 콜 순위, 마사지사 정산 순위, 객실 상태 분포, 노쇼/취소 추이를 그래프로 제공한다.

**검증 조건:**
- 그래프는 원본 기능에서 계산 가능한 데이터만 사용한다.
- CRM, 고객 세그먼트, 마케팅 성과처럼 원본 엑셀에 없는 지표는 v1에서 제공하지 않는다.
- 그래프는 모바일 앱 전용이 아니라 웹 ERP 화면에서 제공된다.

### 4.8 권한과 감사 로그

**설명:** 엑셀 전환의 핵심 개선점은 변경 추적과 마감 후 흔들림 방지다. 지급액에 영향을 주는 모든 주요 변경은 감사 로그로 남긴다.

#### FR-34: 권한 역할

시스템은 관리자, 카운터, 웨이터, 정산 담당, 조회 전용 권한을 지원한다.

**검증 조건:**
- 관리자는 설정, 직원, 수당, 인센, 월마감, 재오픈을 수행할 수 있다.
- 카운터는 콜 입력, 상태 변경, 결제/할인 입력을 수행할 수 있다.
- 웨이터와 조회 전용 사용자는 지정된 조회 화면만 접근할 수 있다.

#### FR-35: 감사 로그

시스템은 민감한 변경의 행위자, 액션, 대상, 변경 전 상태, 변경 후 상태, 시각을 기록한다.

**검증 조건:**
- 콜 상태, 결제/할인, 담당자, 출퇴근, 수당표, 직원, 월마감 확정/취소/재오픈 변경은 감사 로그 대상이다.
- 일반 운영 경로에서 감사 로그는 삭제할 수 없다.
- 감사 로그는 도메인 규칙을 대신 계산하지 않고 변경 이력만 보존한다.

### 4.9 데이터 이관과 검증

**설명:** ERP 1차 버전은 기능 누락 0개를 목표로 한다. 원본 엑셀과 계산 결과를 대조할 수 있는 이관 검증이 필요하다.

#### FR-36: 시트 기능 매핑표

시스템 또는 산출물은 원본 12개 시트와 숨김 시트 `목록`이 어떤 ERP 기능으로 이전됐는지 매핑표를 제공한다.

**검증 조건:**
- `오늘대시보드`, `실시간콜입력`, `웨이터리스트`, `TV현황판`, `운영팀근무인센`, `귀케어일정산`, `마사지사일정산`, `월마감`, `직원DB`, `TV설정`, `설정_코스수당`, `목록`이 모두 포함된다.
- 각 시트는 ERP 화면, 설정, 계산 엔진, 검증 항목 중 하나 이상에 연결된다.
- 숨김 시트 `목록`의 드롭다운 값도 누락 없이 반영된다.

#### FR-37: 계산 대조 검증

시스템은 샘플 데이터 또는 이관 데이터 기준으로 원본 엑셀과 ERP 계산 결과를 대조할 수 있어야 한다.

**검증 조건:**
- 방문완료 기준 결제금액, 할인, 콜인정, 마사지사 수당, 귀케어 풀 계산을 대조한다.
- 웨이터리스트와 TV현황판의 객실 상태, 남은 시간, 종료확인, 색상 표현을 대조한다.
- 운영팀 인센, 귀케어 N분의1 정산, 마사지사 일일정산, 만근수당, 갯수왕 수당, 월마감 최종지급액을 대조한다.

## 5. 화면과 시각화 방향

### 5.1 첫 화면 정보 구조

- 첫 화면은 실시간 객실/콜 현황을 최상단에 둔다.
- 11개 객실 카드는 한눈에 스캔 가능해야 하며, 상태 색상과 남은 시간이 가장 먼저 보여야 한다.
- 오늘 콜/매출/정산 핵심 카드는 객실 카드와 같은 화면에서 보이되, 콜 입력을 방해하지 않아야 한다.
- 상세 분석은 첫 화면 하단 또는 별도 대시보드 영역에서 제공한다.

### 5.2 시각 톤

- 주인이 봤을 때 즉시 좋아할 만큼 선명하고 고급스러운 화면을 목표로 한다.
- 상태 색상은 화려하되 의미가 흔들리지 않아야 한다: `사용중`, `청소중`, `예약`, `종료확인`, `빈방`은 항상 같은 색상 체계를 쓴다.
- 그래프는 장식보다 판단을 돕는 데 집중한다. 숫자, 추세, 순위, 위험 신호가 빠르게 읽혀야 한다.
- 화면 문구는 기능 설명문보다 운영자가 바로 행동할 수 있는 짧은 라벨을 우선한다.

### 5.3 그래프 우선순위

- 1순위 그래프: 일별 매출 추이, 코스별 콜/매출 비중, 객실 상태 분포.
- 2순위 그래프: 마사지사 콜 순위, 마사지사 정산 순위, 노쇼/취소 추이.
- 3순위 그래프: 운영팀 인센 달성률, 월마감 지급 구성.
- 모든 그래프는 원본 엑셀 기능에서 산출 가능한 데이터만 사용한다.

### 5.4 TV 현황판 표시 원칙

- TV 현황판은 입력 없는 전체화면 모드다.
- 멀리서도 객실명, 상태, 남은 시간, 코스, 담당자를 읽을 수 있어야 한다.
- TV 현황판은 객실 현황과 같은 상태 계산 결과를 사용한다.

## 6. 비기능 요구사항

### 6.1 정확성

- ERP 계산 결과는 원본 엑셀의 업무 규칙과 일치해야 한다.
- 행 번호, 셀 좌표, 숨김 행 구조는 구현 기준이 아니라 검증 근거로만 사용한다.
- 월간 집계는 운영월 날짜 조건으로 계산한다.

### 6.2 성능과 실시간성

- 실시간 객실/콜 현황과 TV 현황판은 운영자가 체감할 수 있을 만큼 빠르게 갱신되어야 한다.
- 대시보드 그래프는 조회날짜 또는 운영월 변경 시 즉시 이해 가능한 로딩 상태와 함께 갱신되어야 한다.
- 정산과 월마감 계산은 대량 수식처럼 화면을 멈추게 하지 않고 서버 계산 또는 집계로 처리해야 한다.

### 6.3 보안과 권한

- 인증은 확정 기술스택의 NextAuth/Auth.js를 사용한다.
- 권한별 화면 접근과 기능 실행 범위를 분리한다.
- 지급액, 수당, 마감, 직원 정보에 영향을 주는 기능은 권한 검사를 통과해야 한다.

### 6.4 감사성과 변경 불변성

- 감사 로그는 일반 운영 기능으로 삭제할 수 없다.
- 월마감 확정 스냅샷은 이후 설정 변경으로 자동 재계산되어 흔들리면 안 된다.
- 재오픈은 관리자 권한과 사유 입력을 요구한다.

### 6.5 사용성

- 실시간 객실/콜 현황은 첫 화면에서 주인이 매장 상태를 한눈에 파악할 수 있어야 한다.
- 카운터 입력 화면은 엑셀의 빠른 입력성을 잃지 않아야 한다.
- 정산 화면은 계산 결과와 산출 근거를 함께 보여줘야 한다.

## 7. 제약과 가드레일

- 구현 기술스택은 Next.js + Node.js, PostgreSQL, Prisma, NextAuth/Auth.js를 사용한다.
- UI/UX 기반 기술스택은 Tailwind CSS v4와 shadcn/ui를 사용한다.
- 배포, env, migration 절차는 프로젝트 표준과 동일하게 따른다.
- 원본 엑셀의 기능 보존이 v1 최우선이다.
- 신규 CRM, 마케팅 자동화, 회계 연동, 모바일 앱, 멤버십은 v1 범위가 아니다.
- 이름 문자열은 안정 키로 사용하지 않는다. 직원, 객실, 코스는 고유 ID를 사용한다.
- 현재 수당/인센/가격 정책은 적용월과 스냅샷을 가져야 한다.

## 8. MVP 범위

### 8.1 포함

- 마스터 설정: 운영월, 객실, 직원, 코스, 수당, 인센, 코드, 시간 슬롯
- 예약/콜 원장: 원본 `실시간콜입력` A:S 입력과 계산
- 일별 지출: 원본 `실시간콜입력` U:X 지출 입력

- 웨이터 객실 현황
- TV 현황판
- 오늘/월간 대시보드와 주인용 그래프 시각화
- 마사지사 일일정산
- 귀케어 일일정산
- 운영팀 근무/인센
- 월마감 미리보기, 검토, 확정, 잠금, 재오픈

### 8.2 제외

## 9. 성공 지표

**주요 지표**

- **SM-1: 기능 보존율 100%** — 원본 12개 시트와 숨김 시트 `목록`이 ERP 기능 또는 설정으로 모두 매핑된다. 검증: FR-36.
- **SM-2: 핵심 계산 일치** — 샘플 데이터 기준 방문완료 매출, 할인, 콜인정, 마사지사 수당, 귀케어 풀, 운영팀 인센, 마사지사 정산, 월마감 최종지급액이 원본 엑셀과 일치한다. 검증: FR-12, FR-13, FR-21, FR-23, FR-24, FR-27, FR-28, FR-37.
- **SM-3: 첫 화면 운영 이해도** — 주인이 첫 화면에서 객실 상태, 콜 흐름, 오늘 매출/정산 핵심을 별도 시트 이동 없이 파악할 수 있다. 검증: FR-1, FR-2, FR-3.
- **SM-4: 마감 후 지급액 불변성** — 월마감 확정 후 수당표, 직원명, 원장 변경이 과거 지급 스냅샷을 자동 변경하지 않는다. 검증: FR-29, FR-30, FR-35.

**보조 지표**

- **SM-5: 감사 추적 완전성** — 지급액이나 운영 상태에 영향을 주는 주요 변경이 감사 로그로 남는다. 검증: FR-35.
- **SM-6: 대시보드 시각화 채택** — 주인용 화면에서 일별 매출, 코스별 비중, 마사지사 순위, 객실 상태 분포, 노쇼/취소 추이를 확인할 수 있다. 검증: FR-33.

**카운터 지표**

- **SM-C1: 새 기능 증가를 성공으로 보지 않는다** — v1은 원본 엑셀 기능 보존이 목표이며, CRM/마케팅/멤버십 등 신규 기능 확장은 성공 기준이 아니다.
- **SM-C2: 화려한 시각화가 입력 속도를 해치면 실패다** — 주인용 대시보드는 화려해도 카운터의 콜 입력 흐름을 복잡하게 만들면 안 된다.

## 10. 오픈 질문

1. 월마감 확정 후 재오픈 승인자가 관리자 1인인지, 별도 상위 승인자가 필요한지 운영 정책 확정이 필요하다.
2. TV 현황판 갱신 방식은 자동 새로고침과 실시간 웹소켓 중 구현 단계에서 확정해야 한다.
3. 패키지 매니저, 테스트 러너, lint/format 설정은 구현 착수 전 확정해야 한다.
4. 원본 `sheet.xlsx`의 실제 운영 데이터 이관 범위와 기간을 확정해야 한다.

## 11. 가정 인덱스

- `[ASSUMPTION]` 없음. 사용자가 “모든 내용은 엑셀시트의 기능을 기준으로 하고, 조사하여 선택”하라고 지시했으므로, 문서와 원본 엑셀 분석 근거가 있는 항목은 PRD 결정으로 작성했다.
