## 완료 결과

### Story 5.1: 월마감 미리보기 집계

`listMonthlyClosingPreview()`를 closing orchestration service로 만들고 `/closing` read-only 화면을 구현했다. 마사지사, 운영팀, 귀케어 월 집계는 기존 settlements 서비스를 재사용했고, 화면은 산출 근거와 warning을 함께 표시하도록 정리했다.

### Story 5.2: 만근수당과 갯수왕 수당 계산

마사지사 지급 row에 만근수당, 갯수왕 순위/수당, 최종지급액을 반영했다. Story 4.1 출퇴근 source가 아직 없다는 사실을 조용한 0원 성공으로 숨기지 않고 `missing_story_4_1_source` warning으로 노출했다.

### Story 5.3: 월마감 검토와 확정 스냅샷

`MonthlyClosing` persistence와 `startMonthlyCloseReview()`, `confirmMonthlyClose()`, `getMonthlyClosingSnapshot()`을 도입했다. 확정은 상태 전이, 스냅샷 저장, `monthly_close.confirmed` 감사 로그를 transaction으로 처리하고, 저장된 snapshot을 historical source로 분리했다.

### Story 5.4: 잠금 상태와 지급 영향 데이터 수정 차단

`lockMonthlyClose()`와 공용 `month-lock-guard`를 추가했다. `마감확정`과 `잠금` 운영월 모두 지급 영향 write 차단 상태로 다루며, calls, daily expenses, 귀케어/운영팀 근무상태, 정책 range mutation에 서버 boundary guard를 적용했다.

### Story 5.5: 관리자 사유 기반 재오픈

`reopenMonthlyClose()`를 추가하고 재오픈 권한을 administrator-only `closing:reopen`으로 분리했다. `MonthlyClosing`은 `closeVersion` 기반 versioned snapshot 구조로 변경되어 재오픈 후 재확정해도 이전 snapshot을 보존한다.

### Story 5.6: 월마감 이중확인 모달과 접근성

`마감 확정`의 첫 클릭이 즉시 Server Action을 제출하지 않고 AlertDialog를 열도록 변경했다. 모달은 preview DTO의 지급 요약과 스냅샷 불변 경고를 표시하고, 두 번째 확인에서만 기존 `confirmMonthlyCloseAction()`을 호출한다. `role="alertdialog"`, safe initial focus, focus trap, Esc cancel, focus return을 회귀 대상에 포함했다.

## 잘 된 점

- 월마감 계산과 상태 전이를 UI가 아니라 domain service에 둔 원칙이 끝까지 유지됐다.
- preview와 persisted snapshot의 의미가 분리됐다. `현재 기준 미리보기`, `확정 스냅샷`, `이전 확정 스냅샷` 라벨이 서로 다른 truth를 표현한다.
- Story 5.4에서 `마감확정`도 write 차단 상태로 본 결정이 안전했다. 확정 후 잠금 전 사이에 지급 영향 데이터가 바뀌는 위험을 줄였다.
- Story 5.5에서 `closing:reopen`을 별도 권한으로 분리한 결정이 명확했다. 정산 담당의 확정/잠금 권한과 관리자 재오픈 권한이 섞이지 않았다.
- story별 validator가 scope regression을 계속 잡았다. 예: no UI payout recalculation, snapshot read priority, stale copy, alertdialog contract, E2E fixture parallel-safety.
- Epic 3의 교훈인 "source-of-truth 먼저, 화면은 DTO 표시"가 월마감에도 잘 적용됐다.

## 어려웠던 점

- Story 4.1이 아직 backlog라 만근 인정일 source가 없었다. Epic 5는 이 gap을 계산으로 메우지 않고 warning으로 드러내는 쪽을 선택했다.
- Story 5.3의 단일 snapshot 모델은 Story 5.5의 재오픈/재확정 요구와 충돌했다. 구현 중 versioned snapshot으로 모델을 바꿔야 했다.
- `마감확정`과 `잠금`의 의미가 처음에는 분리되어 있었지만, 지급 영향 write 관점에서는 둘 다 read-only 상태여야 했다. 이 규칙을 calls, settlements, masters, monthly views에 전파하는 작업이 넓었다.
- E2E runtime 검증은 반복적으로 환경 제약에 막혔다. 일부 실행은 sandbox port listen `EPERM`, 이후 실행은 PostgreSQL `ECONNREFUSED`로 browser assertion 전 seed 단계에서 중단됐다.
- Story 5.6은 UI 변경처럼 보이지만 실제 risk는 accessibility contract와 기존 5.3/5.5 confirm flow compatibility였다. 관련 E2E도 함께 수정해야 했다.

## 핵심 학습

1. 월마감은 계산 기능이 아니라 historical truth를 만드는 workflow다. preview, confirmation snapshot, lock, reopen, reconfirm version이 하나의 데이터 생명주기로 설계되어야 한다.
2. `마감확정`과 `잠금`은 화면 상태만이 아니라 지급 영향 write guard의 서버 규칙이다. UI disabled는 보조 신호일 뿐 source of truth가 아니다.
3. 재오픈은 snapshot 삭제가 아니다. 이전 snapshot은 historical reference로 남기고, 재확정은 새 version을 생성해야 감사성과 운영 신뢰를 유지할 수 있다.
4. 후속 dashboard/KPI는 current recalculation과 saved snapshot을 반드시 구분해야 한다. Epic 6은 closing snapshot을 historical source로 먼저 읽고, 재오픈된 `검토중` 월에서는 이전 snapshot을 현재 truth로 쓰지 않아야 한다.
5. runtime E2E가 환경에 막힐 때도 `--list`, unit, validator, Prisma validate, static contract check를 분리 기록해야 한다. 차단된 검증을 통과로 쓰지 않는 습관이 중요하다.

## 다음 Epic 준비: Epic 6 주인용 KPI와 시각화 대시보드

Epic 6은 오늘 KPI, 월간 KPI, 그래프 리포트를 구현한다. Epic 5 결과는 특히 월간 KPI와 월마감 지급 구성 그래프의 source contract가 된다.

필수 준비:

- Dashboard query service가 `closing` snapshot을 historical source로 읽는 규칙을 먼저 고정한다.
- `마감확정`/`잠금` 월의 월간 KPI는 최신 `MonthlyClosing.closeVersion` snapshot을 우선 사용한다.
- 재오픈된 `검토중` 월은 current dashboard query가 현재 truth이고, 이전 snapshot은 historical reference로만 표시한다.
- chart/dashboard UI는 지급액, 수당, 인센 총계를 component에서 재계산하지 않고 dashboard/closing domain DTO를 표시한다.
- 월간 KPI empty state는 snapshot 없음, current preview 없음, policy missing, DB/query failure를 구분해야 한다.

## Significant Discoveries

- Story 5.5가 `MonthlyClosing`을 운영월당 단일 snapshot에서 versioned snapshots로 바꿨다. Epic 6 월간 KPI와 지급 구성 그래프는 이 변경을 전제로 설계해야 한다.
- `마감확정`도 지급 영향 write 차단 상태다. "잠금만 read-only"라고 가정하는 후속 story는 과거 snapshot 불변성을 깨뜨릴 수 있다.
- Story 4.1 만근 source 부재는 월마감 최종지급액의 일부 근거가 warning 상태임을 의미한다. Dashboard는 이 warning을 숨기면 안 된다.

## Readiness Assessment

- Testing and quality: Epic 5 unit tests, story validators, Prisma validation, Playwright `--list` 검증 기록이 있다. Runtime E2E는 환경 제약으로 여러 차례 browser assertion 전에 중단됐다.
- Deployment: repository 내 배포 완료 증거는 확인하지 않았다.
- Stakeholder acceptance: 별도 명시 문서는 확인하지 않았다.
- Technical health: closing domain boundary, versioned snapshot, write guard, permission split, confirm dialog contract는 안정적이다. Dashboard 문서와 project context 일부 문구는 구현 학습 반영이 필요하다.
- Unresolved blockers: Epic 6을 막는 Epic 5 기능 blocker는 발견되지 않았다. 다만 Story 4.1 source gap과 E2E DB connectivity는 후속 검증 리스크다.
