# Story 6.3: 주인용 그래프 리포트

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 주인/관리자,
I want 매출, 코스, 마사지사, 객실, 노쇼/취소 흐름을 그래프로 보기를,
so that 숫자 표만 보지 않고 운영 추세와 위험 신호를 빠르게 판단할 수 있다.

## Acceptance Criteria

1. 주인 또는 관리자가 `/dashboard/reports`에 접근해 운영월과 조회날짜를 선택하면 선택 기준에 맞는 그래프 리포트 DTO가 조회된다. 선택값은 URL search params에 유지되고, 누락/범위 밖 값은 서버에서 canonical redirect로 정규화한다.
2. 그래프 리포트의 모든 집계는 `src/modules/dashboard/dashboard-query-service.ts`의 report query 함수가 소유한다. route/page/chart component는 콜 row, 월마감 snapshot JSON, 정산 evidence, 객실 상태를 직접 재계산하지 않는다.
3. 일별 매출 추이 그래프는 운영월 날짜 범위의 날짜별 방문완료 매출, 순매출, 방문완료 콜 수를 표시한다. 금액은 `calculationStatus === "calculated"`인 `방문완료`/`VISIT_COMPLETE` 콜만 포함하고 예약/사용중/청소중/노쇼/취소는 제외한다.
4. 코스별 콜/매출 비중 그래프는 A~E 코스별 calculated 방문완료 콜 수, 매출 합계, 콜 비중, 매출 비중을 표시한다. 원본 Excel에서 산출 가능한 코스/콜/금액만 사용하고 CRM/고객 세그먼트/마케팅 지표를 추가하지 않는다.
5. 마사지사 담당 콜 순위 그래프는 마사지사1과 마사지사2 담당을 모두 반영하고 `Employee.id` 기준으로 집계한다. 같은 직원이 마사지사1/2에 모두 배정된 경우 담당 evidence 기준을 보존하고 직원명 문자열을 key로 쓰지 않는다.
6. 마사지사 정산 순위 그래프는 정산 domain 결과를 기반으로 월간 정산 금액과 담당 콜을 표시한다. `작성중`/`검토중` 월은 `listMonthlyClosingPreview()` current result, `마감확정`/`잠금` 월은 latest `getMonthlyClosingSnapshot()` snapshot rows를 source로 사용한다.
7. 객실 상태 분포 그래프는 선택 조회날짜의 `listRoomStatuses()` DTO만 사용해 `사용중`, `청소중`, `예약`, `종료확인`, `빈방` 분포를 표시한다. 상태 의미는 기존 `StatusBadge`/status token과 일치해야 하며 색상만으로 의미를 전달하지 않는다.
8. 노쇼/취소 추이 그래프는 운영월 날짜별 노쇼와 취소 건수를 표시한다. 두 series는 텍스트 라벨, 범례, 숫자를 함께 제공하고 매출/순위 series 색상에 객실 상태 색상을 임의 재사용하지 않는다.
9. 운영팀 인센 달성률 또는 월마감 지급 구성 그래프가 표시될 때, 선택 운영월에 운영팀 인센 데이터가 없거나 확정 snapshot이 필요한데 없으면 별도 데이터 없음/미확정/snapshot missing 상태를 표시한다. 존재하지 않는 값을 임의 0 그래프로 꾸미거나 완료 지표처럼 표시하지 않는다.
10. 그래프 UI는 접근 가능한 이름, 텍스트 값, 표/목록 fallback 또는 screen-reader용 데이터 요약을 제공한다. tooltip/legend가 있더라도 색상 단독 전달은 금지한다.
11. `/dashboard/reports`는 dashboard read-only 권한 정책을 따른다. administrator, counter, settlement_manager, read_only_viewer는 접근 가능하고 waiter는 `/rooms`로 redirect된다. mutation UI, Server Action, audit write, monthly close 상태 변경은 추가하지 않는다.
12. loading 중에는 `src/app/(erp)/dashboard/reports/loading.tsx` 또는 동등한 shadcn Skeleton 기반 UI가 레이아웃 윤곽을 유지한다. 조회 실패는 raw server error를 노출하지 않고 안전한 한국어 오류, retry affordance, `role="alert"` semantics를 제공한다.
13. unit/static/e2e 테스트는 URL canonicalization, completed-only 매출 추이, A~E 코스별 콜/매출 비중, 마사지사 담당/정산 순위, 객실 상태 분포, 노쇼/취소 추이, snapshot/current source 분리, 데이터 없음 상태, route access, 색상/라벨 접근성, UI 계산 재구현 금지를 검증한다.

## Dev Notes

- Epic 6 목표는 주인이 오늘/월간 매출, 콜, 객실, 노쇼/취소, 마사지사 순위, 운영팀 인센 흐름을 한눈에 판단하는 것이다. Story 6.3은 그래프 리포트이며 Story 6.4의 공통 로딩/실패/차트 색상 규칙과 겹치지만, 이 story 안에서도 접근 가능한 라벨/숫자/empty state를 구현해야 한다. [Source: `_bmad-output/planning-artifacts/epics.md` Epic 6, Stories 6.3-6.4]
- Story 6.3 원문 AC는 매출 추이, 코스별 콜/매출 비중, 마사지사 콜 순위, 마사지사 정산 순위, 객실 상태 분포, 노쇼/취소 추이, 운영팀 인센/월마감 지급 구성의 데이터 없음 처리를 요구한다. [Source: `_bmad-output/planning-artifacts/epics.md` Story 6.3]
- Architecture는 `dashboard`가 read-only KPI queries를 소유하고, chart library는 dashboard 구현 때 확정하되 dashboard visual이 call-entry grid를 느리게 만들면 안 된다고 정의한다. [Source: `_bmad-output/planning-artifacts/architecture.md` Frontend Architecture; Architectural Boundaries]
- Route 구조 예시는 `/dashboard/reports/page.tsx`를 포함한다. 현재 실제 소스에는 `/dashboard/today`와 `/dashboard/monthly`만 있고 `/dashboard/reports`는 없다. [Source: `_bmad-output/planning-artifacts/architecture.md` Project Organization; Source: `find src/app/(erp)/dashboard`]
- Project context 핵심 규칙: Story 6.1/6.2 기준 dashboard route는 DTO를 소비하고 계산을 화면에서 재구현하지 않는다. Story 6.3도 같은 원칙을 따라야 한다. [Source: `_bmad-output/project-context.md`]
- `getTodayDashboardMetrics()`와 `getMonthlyDashboardMetrics()`는 `src/modules/dashboard/dashboard-query-service.ts`에 있다. Story 6.3은 같은 파일에 report DTO를 추가해 today/monthly DTO를 깨지 않아야 한다. [Source: `src/modules/dashboard/dashboard-query-service.ts`]
- `getDailyCallLedgerSummary()`는 completed calculated rows만 금액/코스 집계에 포함하고 warning counts를 반환한다. 일별 매출 추이는 이 semantics를 재사용해야 한다. [Source: `src/modules/calls/service-call-service.ts`]
- `listServiceCallsForOperatingMonth()`와 `CompletedServiceCallCalculationDto`는 월간 completed calculated 계산 결과를 제공한다. 코스별 매출 비중에 `courseCode`가 필요하면 dashboard service 안에서 computed row DTO를 사용하거나 calls DTO를 명시적으로 확장하고 테스트한다. [Source: `src/modules/calls/service-call-service.ts`]
- `listTherapistDailySettlements()`는 `THERAPIST_1`/`THERAPIST_2` assignment evidence와 `Employee.id`, `staffCode`, `displayName`, `totalCallCount`, `totalCommissionAmount`를 반환한다. 월간 담당 콜 순위는 날짜 반복 또는 월마감 preview rows를 사용하되 직원명 문자열로 묶지 않는다. [Source: `src/modules/settlements/therapist-daily-settlement-service.ts`]
- `listMonthlyClosingPreview()`의 `therapists.rows`는 월간 `totalCallCount`, `monthlySettlementAmount`, full attendance/count king bonus, `finalPayoutAmount`를 포함한다. 정산 순위와 지급 구성은 이 DTO 또는 closing snapshot의 같은 shape를 source로 써야 한다. [Source: `src/modules/closing/monthly-closing-preview-service.ts`]
- `getMonthlyClosingSnapshot()`은 latest versioned snapshot을 반환한다. `마감확정`/`잠금` 월의 지급/정산 그래프는 current preview로 fallback하면 안 된다. [Source: `src/modules/closing/monthly-closing-service.ts`; Source: Story 6.2]
- `listRoomStatuses()`는 current room occupancy와 `종료확인` 판단을 소유한다. 객실 상태 분포 그래프는 이 DTO의 `displayStatus`만 집계하고 active call selection, remaining minutes, expected end를 dashboard에서 재구현하지 않는다. [Source: `src/modules/rooms/room-status-service.ts`]
- `authorization.ts`와 `navigation.ts`는 현재 `/dashboard/today`, `/dashboard/monthly`만 read-only dashboard route로 등록한다. Story 6.3은 `/dashboard/reports`를 같은 정책에 맞춰 추가해야 한다. [Source: `src/lib/authorization.ts`; Source: `src/lib/navigation.ts`]

### Architecture Compliance

- `dashboard`는 read-only query orchestration만 소유한다. 결제/수당/할인/코스/담당자/월마감 확정 로직은 `calls`, `settlements`, `closing`, `rooms` services가 계속 소유한다.
- `/dashboard/reports`는 Server Component 중심으로 구현하고 URL search params를 selection state로 사용한다. Redux/Zustand/TanStack Query를 새로 도입하지 않는다.
- Chart dependency는 범위와 bundle impact가 정당화될 때만 도입한다. 직접 dependency가 없던 Story 6.1/6.2와 달리 6.3은 그래프 리포트이므로 도입 가능하지만, version 추정 금지, exact pin, static validator coverage가 필요하다.
- Stable identifiers는 `OperatingMonth.id`, `Employee.id`, `Course.code`/`Course.id`, `RoomStatusDto.displayStatus`, `MonthlyClosing.id`/`closeVersion`을 사용한다. 직원명, 코스명, 객실명, Excel row/cell을 downstream key로 쓰지 않는다.
- Dashboard report는 mutation UI, Server Action, audit event 생성, monthly close snapshot 생성/수정, payout-impacting write를 만들지 않는다.
- `마감확정`/`잠금` month의 지급/정산 그래프는 latest snapshot을 우선하고 current recalculation과 섞지 않는다. Missing snapshot은 distinct state다.
- 상태 색상은 해당 상태 의미에만 사용한다. 매출, 순위, 정산 series가 `사용중`/`예약`/`청소중`/`종료확인`/`빈방` 색상을 임의 재사용하면 안 된다.
- `shared`에 helper를 추가하려면 다중 caller가 실제로 있어야 한다. 단일 chart formatting/helper는 component local 또는 dashboard module local로 둔다.
