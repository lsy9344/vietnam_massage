# Story 5.4: 잠금 상태와 지급 영향 데이터 수정 차단

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 관리자 또는 정산 담당,
I want 월마감 확정 후 운영월을 잠그고 지급 영향 데이터 수정을 차단하기를,
so that 확정된 월 지급액이 이후 일반 수정으로 흔들리지 않게 할 수 있다.

## Acceptance Criteria

1. 운영월 상태가 `마감확정`이면 administrator 또는 settlement_manager가 `/closing`에서 `잠금`을 실행할 수 있고, 처리 후 운영월 상태는 `잠금`으로 변경된다.
2. 잠금 처리는 `closing` domain service가 소유한다. UI component나 route action은 상태 전이, 감사 로그 payload, snapshot 조회 정책을 직접 구현하지 않는다.
3. 잠금 처리 시 잠금 시각과 행위자가 보존된다. 별도 DB 컬럼을 추가하지 않는 경우에도 `monthly_close.locked` 감사 로그 afterValue에 `lockedAt`, `lockedByAccountId`, 운영월 before/after state가 plain JSON으로 기록되어야 한다.
4. 잠금은 `마감확정 -> 잠금`만 허용한다. `작성중` 또는 `검토중`에서 잠금을 요청하면 `먼저 마감확정이 필요합니다.` 또는 동등한 한국어 오류를 반환하고 status/audit/snapshot 변경을 만들지 않는다.
5. 이미 `잠금`인 운영월에 다시 잠금을 요청하면 중복 잠금은 차단되고 새 audit event가 생성되지 않는다.
6. administrator 또는 settlement_manager가 아닌 사용자의 잠금 요청은 Server Action의 `requirePermission("closing:write")` 또는 더 엄격한 서버 권한 검사에서 차단된다. UI 버튼 숨김/비활성만으로 보안을 대신하지 않는다.
7. 운영월 상태가 `마감확정` 또는 `잠금`인 경우 콜 상태, 결제/할인, 담당자, 일별 지출, 귀케어 근무상태, 운영팀 근무상태처럼 해당 운영월 지급액에 영향을 주는 write mutation은 서버 domain/service boundary에서 차단된다.
8. 잠금 차단 오류 코드는 각 domain에서 기존 `OPERATING_MONTH_LOCKED` 계열을 재사용하거나 동일 의미의 명확한 code로 매핑하고, 사용자 문구는 잠긴/확정된 운영월의 지급 영향 데이터는 수정할 수 없다는 한국어 메시지로 표시한다.
9. 잠긴 운영월을 조회하는 콜 원장, 일별 지출, 귀케어 근무상태, 운영팀 근무상태 화면은 입력 control을 read-only/disabled로 표시하고 잠금 상태 안내를 보여준다. 서버 차단이 source of truth이고 UI는 보조 신호다.
10. 월마감 화면과 월간 지급/KPI 성격의 조회는 `마감확정` 또는 `잠금` 운영월에 확정 스냅샷 값을 `확정 스냅샷`으로 표시하고, 현재 기준 재계산 미리보기와 섞지 않는다.
11. `monthly_close.locked` 감사 로그는 dot notation action 규칙을 통과하고 actor, target operating month/monthly close, before/after state, timestamp를 포함한다. 감사 snapshot에는 Date 객체, `NaN`, class instance를 넣지 않는다.
12. unit/static/e2e 테스트는 확정 후 잠금, 확정 전 잠금 차단, 중복 잠금 차단, 권한 차단, 지급 영향 데이터 write 차단, read-only 표시, 스냅샷 기준 조회, 감사 로그 payload를 검증한다.

## Dev Notes

- Epic 5 목표는 월 지급액을 미리보기, 검토, 확정, 잠금, 재오픈 흐름으로 안전하게 처리하고 확정 스냅샷을 보존하는 것이다. Story 5.4는 Story 5.3이 만든 `MonthlyClosing` snapshot을 보호하는 lock/write-blocking layer다. [Source: `_bmad-output/planning-artifacts/epics.md` Epic 5, Story 5.4]
- FR29는 잠금 상태의 운영월이 일반 사용자의 지급 영향 데이터 수정을 막아야 하고 확정/잠금이 감사 로그 대상이라고 요구한다. [Source: `_bmad-output/planning-artifacts/prds/prd-vietnam_aesthetic-2026-06-07/prd.md` FR-29]
- UX는 `잠금` 상태를 월마감/영향 데이터의 read-only 상태로 정의하고, 확정 스냅샷 기준 조회와 관리자 재오픈만 해제를 명시한다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-vietnam_aesthetic-2026-06-07/EXPERIENCE.md` State Patterns]
- Architecture data flow는 `masters -> calls -> rooms -> settlements -> closing -> dashboard`이고, audit은 민감 변경을 관찰한다. Story 5.4 잠금 규칙은 각 화면 UI가 아니라 domain/service boundary에서 재사용되어야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md` API & Communication Patterns]

### Architecture Compliance

- `src/app` handles form composition and action result display only. Lock state transition, audit payload, and write guard rules belong to domain services.
- `monthly_close.locked` is an audit event, not a second snapshot. The confirmed `MonthlyClosing.snapshotJson` remains immutable.
- Payout-impacting write guards must run server-side after authorization and inside/repeated inside transaction boundaries where writes occur.
- `마감확정` and `잠금` are read-only for payout-impacting writes. This prevents the dangerous gap where a month has a stored snapshot but can still accept edits before a user presses `잠금`.
- Future-month policy changes remain possible. The guard should block overlap with confirmed/locked operating months, not freeze the entire master-data module forever.
- Audit snapshots must be JSON-safe: ISO strings for date/time, number for VND, plain objects only.
- No broad REST API is needed. Use existing Next.js Server Action + domain service pattern.
