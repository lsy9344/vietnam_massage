# Story 4.2: 마사지사 일일정산

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 정산 담당,
I want 방문완료 콜 기준으로 마사지사별 일일 담당 콜과 수당을 계산하기를,
so that 마사지사1과 마사지사2 모두 빠짐없이 당일 정산에 반영할 수 있다.

## Acceptance Criteria

1. 정산 담당자가 `/settlements`의 마사지사 일정산 화면에서 운영월과 날짜를 선택해 조회하면, 선택 날짜의 마사지사별 담당 콜 수와 당일정산 합계가 표시되고 마사지사 식별은 직원 고유 ID 기준으로 처리된다.
2. 선택 날짜의 `방문완료`/`VISIT_COMPLETE` 콜에서 `THERAPIST_1`에 배정된 담당자는 담당 콜로 집계되고, 해당 코스/마사지사/적용월 수당이 당일정산에 반영된다.
3. 선택 날짜의 `방문완료`/`VISIT_COMPLETE` 콜에서 `THERAPIST_2`에 배정된 담당자도 담당 콜로 집계되고, 마사지사2 수당도 별도 담당 기록으로 반영된다.
4. `예약`, `사용중`, `청소중`, `노쇼`, `취소` 및 stable code 동등값의 비완료 콜은 담당 콜 수와 수당 합계에 포함하지 않는다.
5. A~E 코스별 담당 콜 수와 수당 합계를 표시하고, 마사지사1/2 배정 위치와 관계없이 해당 마사지사의 코스별 콜로 집계한다.
6. 적용월 수당 정책 금액이 0원이면 0원 정책으로 표시한다. 적용월 수당 정책을 찾을 수 없으면 해당 담당 건은 0원으로 표시하되 산출 근거에 `정책 없음` warning을 명확히 표시한다.
7. 정산 결과는 금액만 단독으로 보여주지 않고 콜 수, 코스, 담당 역할, 적용 수당 정책 상태를 함께 보여준다.
8. 방문완료/비완료 제외, 마사지사1/2 담당 인정, 코스별 담당 콜, 수당 0원, 정책 누락, 산출 근거를 검증하는 테스트가 통과하며 계산은 UI component가 아니라 `settlements` domain service에서 수행된다.

## Dev Notes

- 이 story는 `src/modules/settlements`가 처음 실제 계산을 소유하는 story다. `calls`가 완료 콜과 담당자별 계산 원천을 제공하고, `settlements`는 마사지사별/코스별 일일 조회용 집계와 산출 근거를 소유한다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements Coverage Validation]
- 현재 `/settlements`는 `ErpEmptyState` placeholder만 렌더링한다. 이 파일을 실제 조회 화면으로 교체하되 `requireRouteAccess("/settlements")`는 유지한다. [Source: src/app/(erp)/settlements/page.tsx]
- 현재 `prisma/schema.prisma`에는 `ServiceCall`, `ServiceCallAssignment`, `CoursePolicy`, `TherapistCourseRate`, `Employee`가 이미 있다. Story 4.2에는 새 persistence model이 필요하지 않다. [Source: prisma/schema.prisma]
- `ServiceCallAssignment.assignmentRole` 값은 `THERAPIST_1`, `THERAPIST_2`, `EARCARE`이고 담당자 저장은 `Employee.id`를 사용한다. 표시명이나 staffCode를 저장 키로 쓰지 않는다. [Source: src/modules/calls/service-call-schema.ts]
- 완료 콜 계산 source of truth는 `src/modules/calls/service-call-service.ts`다. `isCompletedServiceCallStatus()`는 `방문완료`와 `VISIT_COMPLETE`를 완료로 인식하고, 비완료 상태는 `not_completed` 계산으로 제외한다. [Source: src/modules/calls/service-call-service.ts]
- `listCompletedServiceCallCalculationsForDate()`는 현재 `calculationStatus === "calculated"` row만 반환하고 `therapistAssignments[]`에 `role`, `employeeId`, `commissionAmount`를 담는다. Story 4.2의 missing-policy 표시에는 이 helper만으로 부족할 수 있으므로, 필요한 경우 calls-domain helper를 확장하되 UI에서 계산하지 않는다. [Source: src/modules/calls/service-call-service.ts]
- `getDailyCallLedgerSummary()`는 `calculated` 완료 콜만 결제/마사지사정산/귀케어풀/할인/코스 요약에 포함하고 warning count를 별도로 제공한다. Story 4.2도 합계와 warning을 분리해야 한다. [Source: src/modules/calls/service-call-service.ts]
- Story 2.4/2.5 원칙상 `second_therapist_required` invalid completed D-row는 집계에서 제외한다. Story 4.2 정산 화면도 해당 row를 마사지사 지급 합계에 넣지 않는다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 0원 수당 정책은 정상 정책이다. `THR-005`~`THR-050`은 A~E 모두 명시적 0 VND seed가 있을 수 있으므로 missing policy와 0원 정책을 같은 상태로 취급하지 않는다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 정산/월마감 화면은 계산 결과와 산출 근거를 항상 함께 보여줘야 하며 맨숫자 금액만 던지면 안 된다. [Source: _bmad-output/planning-artifacts/ux-designs/ux-vietnam_massage-2026-06-07/EXPERIENCE.md#Voice-and-Tone]
- UI는 한국어 전용, 라이트 모드 전용, 데스크톱 ERP 밀도 기준이다. 랜딩/마케팅형 hero나 decorative card-heavy layout을 만들지 않는다. [Source: _bmad-output/planning-artifacts/ux-designs/ux-vietnam_massage-2026-06-07/EXPERIENCE.md#Surface-Map]
- Server Actions를 새로 만들 경우 별도 `"use server"` 파일과 `ActionResult<T>` 패턴, 권한 재조회, `revalidatePath()`를 기존 calls actions처럼 따른다. 다만 이 story의 기본 범위는 read-only 조회라 mutation action은 만들지 않는다. [Source: src/app/(erp)/calls/actions.ts; Source: https://nextjs.org/docs/app/getting-started/updating-data; Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
