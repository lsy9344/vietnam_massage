# Story 7.3: 이관 검증 리포트와 누락 추적

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 운영자 또는 QA 담당자,
I want 엑셀 기능 매핑과 계산 대조 결과를 리포트로 확인하고 누락/불일치를 추적하기를,
so that v1 기능 보존율 100%와 핵심 계산 일치를 출시 전까지 관리할 수 있다.

## Acceptance Criteria

1. 시트 기능 매핑표와 계산 대조 테스트 결과가 있을 때, 이관 검증 리포트는 시트별 매핑 상태, 계산 대조 상태, 누락 항목, 불일치 항목, 전체 기능 보존율을 표시한다.
2. 원본 12개 시트와 숨김 시트 `목록` 중 매핑되지 않은 항목이 있으면 누락으로 표시하고 기능 보존율 100% 목표를 미달성으로 표시한다.
3. 계산 대조 테스트 또는 fixture 비교에서 불일치가 있으면 불일치 항목, 기대값, ERP 결과값, 관련 FR 또는 story를 표시한다.
4. 계산 대조 테스트가 모두 통과하면 핵심 계산 일치 상태를 통과로 표시하고 방문완료 매출, 할인, 콜인정, 수당, 귀케어 풀, 운영팀 인센, 마사지사 정산, 월마감 최종지급액 대조를 포함한다.
5. 누락 또는 불일치 항목은 `미확인`, `수정중`, `재검증 필요`, `통과` 상태로 관리할 수 있고, 상태 변경 이력 또는 담당자 메모가 남는다.
6. 특정 시트, FR, epic/story, 상태 기준으로 필터링하면 관련 매핑과 대조 결과만 표시되어 출시 전 남은 위험을 빠르게 확인할 수 있다.
7. 숨김 시트 `목록`의 드롭다운 값 또는 시트별 계산 항목이 누락되어 있으면 리포트는 기능 보존율을 100%로 표시하지 않는다.
8. 매핑 완료, 누락 항목, 계산 불일치, 전체 통과, 필터링, 상태 추적 테스트가 통과하고, 신규 CRM/마케팅/멤버십 같은 v1 제외 범위는 성공 기준에 포함하지 않는다.

### Data Model Guidance

- Suggested models:
  - `MigrationVerificationIssue`: `id`, `itemKey`, `kind`, `sourceSheet`, `relatedRequirement`, `relatedStory`, `status`, `assigneeName`, `note`, `createdAt`, `updatedAt`.
  - `MigrationVerificationIssueHistory`: `id`, `issueId`, `previousStatus`, `newStatus`, `note`, `changedByAccountId`, `changedAt`, `createdAt`.
- Suggested DB table names: `migration_verification_issues`, `migration_verification_issue_histories`.
- `itemKey` should be unique. `kind` should be bounded, for example `sheet_mapping`, `calculation_comparison`, `manual_risk`.
- Store expected/actual values in report DTO, not necessarily in issue rows. Persisted issue rows track workflow state; current evidence should still come from mapping/fixture sources so stale copied evidence does not drift.
- If implementation chooses a different persistence shape, it must still support deterministic item keys, four allowed statuses, notes/history, and admin-only mutation.

### Anti-Reinvention Guardrails

- Do not create a second sheet mapping source. Use `SHEET_FEATURE_MAPPINGS`.
- Do not create a second calculation engine in `src/modules/migration`. Use Story 7.2 fixture/test results and existing domain service contracts.
- Do not parse `sheet.xlsx`, Excel formulas, workbook ranges, or cell coordinates at runtime. Excel references remain evidence strings only.
- Do not add chart dependencies for this report. Tables, summary tiles, badges, and filters are enough.
- Do not broaden `read_only_viewer` to `/masters/*`.
- Do not use audit logs as the QA tracking storage. Audit remains for sensitive operational changes; migration issue history is its own domain.
- Do not include out-of-scope v1 exclusions such as CRM, marketing automation, accounting integration, mobile app, or membership in preservation success metrics.

### UI/UX Requirements

- This is an operational QA screen, not a landing page. Keep it dense, scannable, and table-first.
- Summary tiles should answer: preservation rate, missing sheets, calculation pass/mismatch, open tracking items.
- Filter controls should be compact and URL-backed so QA can share a filtered view.
- Status labels must include text, not color only. Use existing tokens/classes and `StatusBadge` only where it represents existing call/room status semantics; QA tracking statuses can use neutral/danger/success text badges without reusing room status colors incorrectly.
- Admin mutation feedback must be inline Korean `role="status"` or `role="alert"` and must not expose raw server errors.
