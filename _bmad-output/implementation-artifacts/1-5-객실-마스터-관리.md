# Story 1.5: 객실 마스터 관리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want 11개 객실을 고유 ID와 표준 표시명으로 관리하기를,
so that 콜 원장, 객실 현황, TV 현황판이 객실명을 안정적으로 참조하고 표시할 수 있다.

## Acceptance Criteria

1. **기본 11개 객실 목록**
   - **Given** 관리자가 객실 마스터 화면에 접근한다
   - **When** 초기 객실 목록을 조회한다
   - **Then** 기본 객실 `101 호실`, `102 호실`, `103 호실`, `201 호실`, `202 호실`, `203 호실`, `301 호실`, `302 호실`, `303 호실`, `401 호실`, `402 호실`이 표시된다
   - **And** 각 객실은 표시명과 별도의 고유 ID를 가진다.

2. **표시명 수정과 고유 ID 보존**
   - **Given** 관리자가 객실 표시명을 수정한다
   - **When** 저장한다
   - **Then** 표시명 변경은 저장된다
   - **And** 객실 고유 ID는 변경되지 않는다.

3. **숨김 목록 이관 참조값 보존**
   - **Given** 숨김 시트 `목록`의 `1번방` 형식 이관 참조값이 있다
   - **When** 객실 마스터가 표시된다
   - **Then** 운영 표준 표시명은 `101 호실` 형식을 사용한다
   - **And** `1번방` 형식은 이관/검증 참조값으로만 보존된다.

4. **삭제 대신 비활성 처리**
   - **Given** 객실이 콜 원장 또는 객실 현황에서 참조될 수 있다
   - **When** 관리자가 객실을 삭제하려 한다
   - **Then** 물리 삭제 대신 비활성 처리를 사용한다
   - **And** 과거 콜/정산/현황 이력의 객실 참조는 깨지지 않는다.

5. **표시 순서 관리**
   - **Given** 객실 순서가 TV 현황판과 객실 현황에 필요하다
   - **When** 관리자가 객실 표시 순서를 확인하거나 수정한다
   - **Then** 객실은 정렬 순서를 가진다
   - **And** 이후 객실 카드/TV 화면은 이 순서를 사용할 수 있다.

6. **감사 로그 기록**
   - **Given** 객실 정보가 생성, 수정, 비활성 처리된다
   - **When** 저장이 완료된다
   - **Then** 행위자, 대상 객실, 변경 전후값, 시각이 감사 로그로 기록된다
   - **And** 감사 로그는 객실 상태 계산을 대신하지 않고 변경 이력만 보존한다.

7. **검증**
   - **Given** 개발자가 객실 마스터를 검증한다
   - **When** 기본 11개 객실 seed, 표시명 수정, 비활성 처리, 감사 로그 테스트를 실행한다
   - **Then** 모든 테스트가 통과한다
   - **And** 객실 식별은 표시명이 아니라 고유 ID 기준임이 검증된다.

### Architecture Guardrails

- Keep route composition in `src/app/(erp)/masters/rooms`, domain logic in `src/modules/masters`, authorization in `src/lib`, and immutable change history in `src/modules/audit`. Do not put room seed, sort, deletion, or audit rules in React components. [Source: `_bmad-output/planning-artifacts/architecture.md` - Project Organization, Service Boundaries]
- Prisma model names use PascalCase singular and DB tables use snake_case plural with `@@map`. For this story use `Room` mapped to `rooms`. [Source: `_bmad-output/planning-artifacts/architecture.md` - Naming Patterns]
- Data exchange uses camelCase field names. Display names are mutable values; stable references use DB `id`. Do not use `displayName`, `migrationReferenceName`, Excel row order, or sheet labels as foreign keys. [Source: `_bmad-output/planning-artifacts/architecture.md` - Data Exchange Formats; `_bmad-output/project-context.md` - critical rules]
- Server Actions are reachable by direct POST, so every room mutation action must authenticate, authorize, validate, call a domain service, and map safe Korean errors. Use the existing `ActionResult<T>` shape. [Source: `_bmad-output/planning-artifacts/architecture.md` - API & Communication Patterns; External: Next.js Forms - `https://nextjs.org/docs/app/guides/forms`; External: Next.js Mutating Data - `https://nextjs.org/docs/app/getting-started/mutating-data`]
- Zod 4 is the validation baseline. Use `safeParse()` and `z.enum()`/refinements where fixed values or constrained inputs are needed. [External: Zod API - `https://zod.dev/api`]
- Prisma 7 schema is configured with `prisma.config.ts`; PostgreSQL native date/table/field mapping must follow the current schema style. Prisma docs support singular PascalCase models, `@map`/`@@map`, and PostgreSQL native type attributes such as `@db.Date`; for this story no date-only room fields are required. [External: Prisma Schema API - `https://www.prisma.io/docs/orm/reference/prisma-schema-reference`]
