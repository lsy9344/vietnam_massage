# Story 1.3: 감사 로그 기반과 조회 화면

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want 지급액이나 운영 상태에 영향을 주는 변경 이력을 감사 로그로 남기고 조회하기를,
so that 엑셀 전환 후 누가 무엇을 바꿨는지 추적하고 과거 정산의 신뢰성을 지킬 수 있다.

## Acceptance Criteria

1. **감사 이벤트 기록**
   - **Given** 감사 로그가 필요한 도메인 변경이 발생한다
   - **When** domain service 또는 Server Action이 변경을 저장한다
   - **Then** 감사 로그는 actor ID, action name, target type, target ID, before value, after value, timestamp를 기록한다
   - **And** 필요한 경우 reason을 함께 기록할 수 있다.

2. **dot notation action name**
   - **Given** 감사 이벤트 이름이 기록된다
   - **When** 이벤트가 저장된다
   - **Then** action name은 `service_call.status_changed`, `monthly_close.confirmed` 같은 dot notation을 사용한다
   - **And** 이벤트 이름은 도메인 계산을 대신하지 않고 변경 이력만 설명한다.

3. **삭제 불가**
   - **Given** 일반 운영 사용자가 감사 로그를 삭제하려 한다
   - **When** 삭제 요청이 발생한다
   - **Then** 일반 운영 경로에서는 감사 로그 삭제가 허용되지 않는다
   - **And** 감사 로그는 불변 이력으로 보존된다.

4. **관리자 조회 화면**
   - **Given** 관리자가 감사 로그 화면에 접근한다
   - **When** 감사 로그 목록을 조회한다
   - **Then** 행위자, 액션, 대상, 변경 전후값, 시각이 표시된다
   - **And** 지급액 영향 변경을 빠르게 찾을 수 있도록 대상 유형 또는 기간 기준 조회를 지원한다.

5. **권한 차단**
   - **Given** 권한이 없는 사용자가 감사 로그 화면에 접근한다
   - **When** route 또는 Server Action이 요청을 처리한다
   - **Then** 접근은 서버에서 차단된다
   - **And** sidebar에서 감사 로그 그룹은 허용 역할에게만 표시된다.

6. **후속 story 재사용성**
   - **Given** 이후 마스터, 콜 원장, 정산, 월마감 story가 변경 이력을 남겨야 한다
   - **When** 해당 story가 감사 로그 서비스를 호출한다
   - **Then** 공통 감사 로그 서비스와 payload 형식을 재사용할 수 있다
   - **And** 각 story는 자체 도메인 변경 전후값만 제공하면 된다.

7. **검증**
   - **Given** 개발자가 감사 로그 기반을 검증한다
   - **When** 감사 이벤트 생성, 조회, 권한 차단, 삭제 불가 테스트를 실행한다
   - **Then** 모든 테스트가 통과한다
   - **And** 후속 story에서 감사 로그 기록을 미래 의존 없이 사용할 수 있다.

### Architecture Guardrails

- Keep route composition in `src/app`, reusable ERP UI in `src/components/domain`, domain services in `src/modules/audit`, and cross-cutting permission helpers in `src/lib`. Do not put audit persistence logic in React components. [Source: `_bmad-output/planning-artifacts/architecture.md` - Project Organization]
- Prisma model names use PascalCase singular and DB tables use snake_case plural with `@@map`. For this story that means `AuditLog` mapped to `audit_logs`. [Source: `_bmad-output/planning-artifacts/architecture.md` - Naming Patterns]
- Server Actions and Server Functions are public-facing server entrypoints. Next.js current docs say to verify authentication and authorization inside every Server Function; do not rely on sidebar hiding or route layout alone for audit queries. [External: Next.js Mutating Data - `https://nextjs.org/docs/app/getting-started/mutating-data`]
- Existing architecture requires Server Actions/Route Handlers to authenticate, authorize, validate with Zod when accepting input, call domain services, and return `ActionResult<T>` for user-visible mutation results. [Source: `_bmad-output/planning-artifacts/architecture.md` - API & Communication Patterns, Action Response Format]
- Prisma official docs support `Json` fields for unstructured JSON values; PostgreSQL maps Prisma `Json` to `jsonb`. Use this for before/after snapshots rather than ad hoc stringification. [External: Prisma JSON fields - `https://docs.prisma.io/docs/orm/prisma-client/special-fields-and-types/working-with-json-fields`; External: Prisma PostgreSQL connector - `https://docs.prisma.io/docs/orm/overview/databases/postgresql`]
- NextAuth v4 Credentials/JWT/session callback model remains the project baseline. Story 1.3 should not change auth provider strategy. [External: NextAuth Credentials Provider - `https://next-auth.js.org/providers/credentials`; External: NextAuth Callbacks - `https://next-auth.js.org/configuration/callbacks`; Source: `_bmad-output/project-context.md` - technology stack]

### Anti-Patterns to Avoid

- Do not implement audit as a generic free-form activity feed that captures non-sensitive UI visits. Story 1.3 covers payout/operational-state impacting changes and administrator review.
- Do not create audit update/delete UI, delete Server Action, or hard-delete helper in `audit-service.ts`.
- Do not let `audit` calculate payouts, call status, monthly close state, employee status, or permissions. It only records changes after the owning domain validates and writes.
- Do not store `beforeValue`/`afterValue` as display-only strings. Store structured JSON so future close review and traceability reports can inspect fields.
- Do not trust `session.user.role` for audit access. Use `requirePermission("audit:read")` or DB-backed account authorization.
- Do not add public REST endpoints for audit browse unless an external integration enters scope. Internal ERP browse can use Server Component queries or Server Actions.
- Do not change role landing/sidebar rules from Story 1.2 while adding audit.
- Do not create fake business audit rows in production code. Test fixtures/seeds must be clearly test-local.

### Testing Standards

- Minimum static validation: `npm run lint` must pass and include Story 1.1 + Story 1.2 + Story 1.3 checks.
- Static checks should cover schema, service names, dot notation validation, deletion absence, audit permission use, and placeholder removal.
- E2E should verify administrator `/audit` visibility and non-admin `/audit` direct-route blocking/sidebar hiding.
- If a domain/unit test runner is not available, keep service logic simple and validate critical contracts through the static validator until the project adds a unit runner.
- Manual acceptance checks:
  - Administrator can open `/audit` and see a table/list with time, actor, action, target, before/after, reason.
  - Target type and period filters change the query without blanking the screen.
  - Counter, settlement manager, waiter, and read-only viewer do not see 감사 로그 in sidebar and direct `/audit` redirects to their landing.
  - `recordAuditEvent()` rejects non-dot-notation actions.
  - There is no app-level path to delete an audit log.
  - Future story service code can call `recordAuditEvent()` by passing only actor/action/target/before/after/reason.
