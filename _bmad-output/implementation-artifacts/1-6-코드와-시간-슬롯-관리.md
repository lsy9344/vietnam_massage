# Story 1.6: 코드와 시간 슬롯 관리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want 상태, 결제수단, 할인구분, 근무상태, 확인값, 시간 슬롯 코드를 관리하기를,
so that 콜 원장과 정산 화면이 원본 엑셀의 드롭다운 값을 누락 없이 사용할 수 있다.

## Acceptance Criteria

1. **상태 코드 기본값**
   - **Given** 관리자가 코드 관리 화면에 접근한다
   - **When** 상태 코드 목록을 조회한다
   - **Then** `예약`, `사용중`, `청소중`, `방문완료`, `노쇼`, `취소`가 포함된다
   - **And** 상태 표시값은 한국어 원문 그대로 보존된다.

2. **결제수단 코드 기본값**
   - **Given** 관리자가 결제수단 코드 목록을 조회한다
   - **When** 목록이 표시된다
   - **Then** `현금`, `카드`, `계좌`, `기타`가 포함된다
   - **And** 비활성 처리는 가능하지만 과거 콜 참조값은 깨지지 않는다.

3. **할인구분 코드 기본값**
   - **Given** 관리자가 할인구분 코드 목록을 조회한다
   - **When** 목록이 표시된다
   - **Then** `일주일내방문`, `생일자`, `후기작성`이 포함된다
   - **And** 할인금액 다양화는 v1 범위에 포함하지 않는다.

4. **근무상태 코드 기본값**
   - **Given** 관리자가 근무상태 코드 목록을 조회한다
   - **When** 목록이 표시된다
   - **Then** `정상`, `휴무`, `지각`, `조퇴`, `결근`이 포함된다
   - **And** 정산 story에서 정상 상태만 지급 대상 판정에 사용할 수 있다.

5. **확인값 코드 기본값**
   - **Given** 관리자가 확인값 코드 목록을 조회한다
   - **When** 목록이 표시된다
   - **Then** 원본 숨김 시트 `목록`의 `확인` 값인 `Y`, `N`이 포함된다
   - **And** 이후 콜 원장 확인값 입력에서 빈 값과 `Y`/`N`을 구분할 수 있다.

6. **시간 슬롯 기본값**
   - **Given** 관리자가 시간 슬롯 목록을 조회한다
   - **When** 입력 시간 슬롯이 표시된다
   - **Then** `11:00`부터 `01:00`까지 30분 단위 29개 슬롯이 제공된다
   - **And** 자정을 넘는 `00:00`, `00:30`, `01:00` 슬롯도 포함된다.

7. **삭제 대신 비활성 처리**
   - **Given** 코드 또는 시간 슬롯이 콜 원장, 정산, 객실 현황에서 참조될 수 있다
   - **When** 관리자가 값을 삭제하려 한다
   - **Then** 물리 삭제 대신 비활성 처리를 사용한다
   - **And** 과거 데이터의 코드 참조는 유지된다.

8. **감사 로그와 검증**
   - **Given** 코드 값이나 시간 슬롯이 생성, 수정, 비활성 처리된다
   - **When** 저장이 완료된다
   - **Then** 행위자, 코드 유형, 변경 전후값, 시각이 감사 로그로 기록된다
   - **And** 기본 코드 seed, 시간 슬롯 생성, 비활성 처리, 감사 로그 테스트가 통과한다.

### Epic 1 Context

Epic 1 establishes ERP 운영 기준과 접근 통제 before employee, course, rate, incentive, and call-ledger work. Story 1.6 is the shared dropdown/reference-data story for later `calls`, `settlements`, `rooms`, and TV/dashboard surfaces. [Source: `_bmad-output/planning-artifacts/epics.md` - Epic 1, Story 1.6]

FR9 requires administrators to manage status, payment method, discount type, confirmation value, attendance state, and time slot codes. These values feed FR10 call input, FR11 status changes, FR13 discount calculation, FR22/FR24 attendance settlement rules, and FR16/FR18 room status display. [Source: `_bmad-output/planning-artifacts/epics.md` - FR9-FR18, FR22, FR24]

This story must not implement call-ledger autosave, service-call status history, payment amount calculation, fixed 100,000 discount calculation, D-course validation, room-status latest-call lookup, or settlement payout logic. It only supplies stable reference data and audit-visible master mutations. [Source: `_bmad-output/planning-artifacts/epics.md` - Story 1.6, Story 2.1-2.3, Story 3.1]
