# Story 3.2: 첫 화면 실시간 객실/콜 현황

Status: done

## Story

As a 주인/관리자,  
I want 로그인 후 첫 화면에서 객실 상태와 오늘 콜/매출 요약을 한눈에 보기를,  
so that 별도 엑셀 시트를 이동하지 않고 매장 운영 상태를 즉시 판단할 수 있다.

## Acceptance Criteria

1. **Given** 주인 또는 관리자가 로그인한다  
   **When** ERP에 진입한다  
   **Then** 첫 화면은 실시간 객실/콜 현황으로 표시된다  
   **And** 11개 객실 카드가 화면 최상단의 4열 그리드로 표시된다.
2. **Given** 첫 화면 객실 카드가 렌더링된다  
   **When** 객실 상태 데이터가 조회된다  
   **Then** 각 카드는 Story 3.1의 `RoomStatusDto`를 사용한다  
   **And** 화면 component는 객실 상태 계산을 직접 재구현하지 않는다.
3. **Given** 객실 카드가 표시된다  
   **When** 각 객실 상태가 `사용중`, `청소중`, `예약`, `종료확인`, `빈방` 중 하나다  
   **Then** 카드에는 상태별 색상, 텍스트 라벨, 글리프가 함께 표시된다  
   **And** 코스, 담당자, 시작시간, 남은분, 종료예정이 가능한 범위에서 표시된다.
4. **Given** 표시 상태가 `종료확인`인 객실이 있다  
   **When** 첫 화면이 렌더링된다  
   **Then** 해당 객실은 주의 상태로 구분되어 보인다  
   **And** "결제·확인 필요" 또는 동등한 짧은 한국어 행동 라벨이 표시된다.
5. **Given** 오늘 콜 원장 데이터가 있다  
   **When** 첫 화면 요약이 조회된다  
   **Then** 오늘 예약, 사용중, 청소중, 방문완료, 노쇼, 취소 수가 표시된다  
   **And** 매출/정산 관련 지표는 `방문완료` 기준만 포함한다.
6. **Given** 오늘 결제와 코스 데이터가 있다  
   **When** KPI 영역이 렌더링된다  
   **Then** 오늘 결제합계, 순매출, 코스별 방문완료 수가 표시된다  
   **And** 지출이 있는 경우 순매출은 일별 지출 요약 기준을 반영한다.
7. **Given** 첫 화면 데이터가 로딩 중이다  
   **When** 사용자가 화면에 진입한다  
   **Then** shadcn Skeleton 또는 명확한 로딩 상태가 표시된다  
   **And** blank screen으로 보이지 않는다.
8. **Given** 첫 화면 데이터 갱신이 지연된다  
   **When** 마지막 갱신 이후 지연이 감지된다  
   **Then** 마지막 갱신 시각 또는 갱신 지연 상태가 표시된다  
   **And** 사용자가 오래된 상태를 최신 상태로 오해하지 않게 한다.
9. **Given** 개발자가 첫 화면을 검증한다  
   **When** 11개 객실 카드, 오늘 상태 요약, 방문완료 기준 KPI, 종료확인 카드, 로딩/지연 상태 테스트를 실행한다  
   **Then** 모든 테스트가 통과한다  
   **And** 첫 화면은 콜 원장 입력 기능 없이 조회 중심 화면으로 동작한다.

### Data and Calculation Rules

- 객실 카드 데이터 source of truth는 `listRoomStatuses({ operatingMonthId, serviceDate })`다. page/component에서 활성 콜 선택, 자정 넘김, 종료확인, 남은분 계산을 다시 구현하지 않는다. [Source: `_bmad-output/project-context.md` - Story 3.1 rules]
- 상태 요약 source of truth는 `getDailyCallLedgerSummary()`다. 예약/사용중/청소중/방문완료/노쇼/취소 카운트는 콜 원장 행 상태 기준이어야 한다. 11개 객실 현재 상태를 집계하면 하루 콜 원장 summary와 달라질 수 있다.
- 매출/정산 KPI는 `calculationStatus === "calculated"`인 `방문완료` row만 포함한다. 정책 누락, 수당 누락, D코스 마사지사2 누락 row는 금액/코스별 요약에서 제외한다. [Source: `src/modules/calls/service-call-service.ts` - `getDailyCallLedgerSummary()`]
- 순매출 공식은 `netSales = paymentTotal - expenseTotal`이고 active `DailyExpense`만 지출합계에 포함한다. [Source: `_bmad-output/project-context.md` - Story 2.5 rules]
- 코스별 방문완료 수는 mutable course label이 아니라 stable `Course.code` A-E 기준 summary를 쓴다. [Source: `_bmad-output/project-context.md` - Story 2.5 rules]

### Architecture Compliance

- App Router page는 기본적으로 Server Component이므로 `/live/page.tsx`에서 서버 측으로 Prisma-backed domain services를 직접 호출할 수 있다. 인증/권한은 `requireRouteAccess("/live")`로 먼저 처리한다. [Source: Next.js App Router data fetching docs, https://nextjs.org/docs/app/getting-started/fetching-data]
- Client Component는 browser-only 갱신 제어(`setInterval`, `router.refresh()`, local pending state)에만 쓴다. domain query나 권한 검사를 Client Component로 옮기지 않는다. [Source: Next.js Server/Client Components docs, https://nextjs.org/docs/app/building-your-application/rendering/server-components]
- Server Actions는 mutation이 필요한 경우에만 추가한다. 이번 story는 읽기 전용이므로 broad REST API나 write action이 필요 없다. [Source: `_bmad-output/planning-artifacts/architecture.md` - Internal Communication]
- polling은 v1에서 시작 가능한 실시간성 방식이다. `@tanstack/react-query`는 현재 package에 설치되어 있지 않으므로 이 story에서는 Next `router.refresh()` 기반으로 충분하다. 나중에 React Query를 도입하면 `refetchInterval`은 주기 polling을 지원한다. [Source: TanStack Query polling docs, https://tanstack.com/query/latest/docs/react/guides/polling]
