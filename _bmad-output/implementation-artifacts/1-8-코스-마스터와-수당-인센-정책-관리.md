# Story 1.8: 코스 마스터와 수당/인센 정책 관리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want 코스, 마사지사 수당, 운영팀 인센 정책을 적용월 이력으로 관리하기를,
so that 콜 계산과 정산이 현재 정책을 쓰면서도 과거 지급액은 흔들리지 않게 할 수 있다.

## Acceptance Criteria

1. **기본 코스 마스터**
   - **Given** 관리자가 코스 마스터 화면에 접근한다
   - **When** 기본 코스 목록을 조회한다
   - **Then** A 60분, B 90분, C 90분, D 90분 2:1, E 120분 코스가 표시된다
   - **And** 각 코스는 고유 ID, 코드, 이름, 시간, 기본판매가, 운영팀 콜인정, 귀케어 풀/콜, 마사지사2 필요 여부, TV 표시명, 활성 여부를 가진다.

2. **D코스 2인 설정**
   - **Given** 관리자가 D코스를 조회한다
   - **When** 코스 상세가 표시된다
   - **Then** D코스는 `마사지사2 필요`가 Y로 관리된다
   - **And** Story 2.4의 D코스 검증은 코스명 문자열이 아니라 `Course.id`와 `requiresSecondTherapist` 정책 설정을 사용할 수 있다.

3. **코스 표시/가격 정책 이력**
   - **Given** 관리자가 코스 가격 또는 표시 정보를 변경한다
   - **When** 적용 시작월과 선택적 적용 종료월을 지정해 저장한다
   - **Then** 변경값은 적용월 범위를 가진 정책 이력으로 저장된다
   - **And** 표시명 또는 TV 표시명 변경은 코스 고유 ID와 코스 코드를 변경하지 않는다.

4. **마사지사 개인별 코스 수당 정책**
   - **Given** 관리자가 마사지사 개인별 코스 수당을 관리한다
   - **When** 마사지사, 코스, 금액, 적용 시작월, 적용 종료월을 저장한다
   - **Then** 수당 정책은 해당 적용월 범위에서만 조회된다
   - **And** 기본 seed는 마사지사1~4의 A/B/C 수당 700000/900000/900000과 D/E 0, 마사지사5~50의 A~E 수당 0을 원본 이관값으로 보존하고 수정 가능하다.

5. **운영팀 일일 인센 정책**
   - **Given** 관리자가 운영팀 일일 인센 정책을 조회한다
   - **When** 정책이 표시된다
   - **Then** 일 총콜 30/40/50 기준과 개인별 50000/100000/200000 지급액이 보존된다
   - **And** 정책은 적용월 이력을 가진다.

6. **운영팀 월 인센 정책**
   - **Given** 관리자가 운영팀 월 인센 정책을 조회한다
   - **When** 정책이 표시된다
   - **Then** 월 총콜 1000/1100/1200/1300/1400/1500 구간과 전체 월인센 3000000/5000000/8000000/12000000/18000000/25000000이 보존된다
   - **And** 팀장 30%, 카운터팀 35%, 웨이터팀 35% 분배율과 적용월 이력을 가진다.

7. **정책 변경과 과거 스냅샷 불변성**
   - **Given** 코스, 수당, 인센 정책이 과거 월마감 스냅샷에 영향을 줄 수 있다
   - **When** 관리자가 정책을 변경한다
   - **Then** 현재/미래 계산에 사용할 새 정책 이력이 생성되거나 기존 정책 종료월이 조정된다
   - **And** 확정된 과거 월마감 스냅샷은 자동 재계산되지 않는다.

8. **정책 감사 로그**
   - **Given** 코스, 수당, 인센 정책이 생성, 수정, 비활성 처리된다
   - **When** 저장이 완료된다
   - **Then** 행위자, 대상 코스 또는 정책, 변경 전후값, 적용월, 시각이 감사 로그로 기록된다
   - **And** 지급액에 영향을 주는 변경으로 분류된다.

9. **검증**
   - **Given** 개발자가 정책 관리 기능을 검증한다
   - **When** 기본 코스 seed, D코스 2인 설정, 수당 0원 초기값, 일/월 인센 정책, 적용월 이력, 감사 로그 테스트를 실행한다
   - **Then** 모든 테스트가 통과한다
   - **And** 콜 계산과 정산 story에서 정책 조회 서비스를 재사용할 수 있다.

### Architecture Guardrails

- `src/app` owns route composition and page rendering. Policy business rules belong in `src/modules/masters`, not in React components. [Source: `_bmad-output/planning-artifacts/architecture.md` - Project Organization]
- Prisma model names use PascalCase singular and DB tables/columns use snake_case mapping. Index/unique names should follow `idx_*`/`uq_*`. [Source: `_bmad-output/planning-artifacts/architecture.md` - Naming Patterns]
- Server Actions are thin mutation adapters: authenticate, authorize, validate, call domain service, map result. [Source: `_bmad-output/planning-artifacts/architecture.md` - API & Communication Patterns]
- Server Action responses must use `ActionResult<T>` with `ok`, `data`, `fieldErrors`, `formError`, and optional `domainErrorCode`. [Source: `_bmad-output/planning-artifacts/architecture.md` - Action Response Format; `src/lib/action-result.ts`]
- Money values are whole VND integers, not formatted display strings. Service dates/months are ISO strings. [Source: `_bmad-output/planning-artifacts/architecture.md` - Data Exchange Formats]
- Policy records need effective-month history, and monthly close snapshots must not be overwritten by current policy changes. [Source: `_bmad-output/planning-artifacts/architecture.md` - Data Boundaries]
- Audit action names must use dot notation and snapshots must be JSON serializable plain values. `recordAuditEvent()` rejects Date objects, NaN, functions, and class instances. [Source: `src/modules/audit/audit-event.ts`; `src/modules/audit/audit-service.ts`; `_bmad-output/project-context.md`]
- Next.js Server Actions can revalidate changed views with `revalidatePath()`. Keep `revalidatePath("/masters/courses")` after successful policy writes. [External: Next.js Server Actions and Mutations docs]
- Prisma relations should model the database relationships explicitly, and mapped constraint/index names are supported by Prisma schema attributes. [External: Prisma relations and indexes docs]
- Zod 4 is stable in the project (`zod@4.1.12`); use existing `safeParse()` + flattened field-error pattern from prior stories. [External: Zod docs; Source: `package.json`]
