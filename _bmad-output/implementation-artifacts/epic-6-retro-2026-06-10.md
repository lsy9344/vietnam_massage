## 완료 결과

### Story 6.1: 오늘 KPI 대시보드

`getTodayDashboardMetrics()`와 `/dashboard/today`를 구현했다. 오늘 KPI는 `getDailyCallLedgerSummary()`와 `listTherapistDailySettlements()`를 재사용하고, route component가 상태 수, 금액, 코스 완료, 마사지사 evidence를 재계산하지 않도록 정리했다.

### Story 6.2: 월간 KPI 대시보드와 스냅샷 조회

`getMonthlyDashboardMetrics()`와 `/dashboard/monthly`를 구현했다. `sourceBasis`로 `current_recalculation`, `closed_snapshot`, `snapshot_missing`을 구분했고, `마감확정`/`잠금` 월은 latest `MonthlyClosing` snapshot을 우선 source로 사용하며 missing snapshot을 current preview로 숨기지 않도록 고정했다.

### Story 6.3: 주인용 그래프 리포트

`getDashboardGraphReport()`와 `/dashboard/reports`를 구현했다. 일별 매출, 코스 비중, 마사지사 콜/정산 순위, 객실 상태 분포, 노쇼/취소 추이, 지급 구성은 dashboard DTO에서 집계하고 화면은 SVG/CSS presentation primitive와 fallback table/list만 담당한다.

### Story 6.4: 대시보드 로딩/실패/차트 색상 규칙

Dashboard loading/error/empty/chart color contract를 명시하고 static validator를 추가했다. missing calculated completed call 데이터를 성공적인 0값 그래프로 보이지 않도록 `/dashboard/reports`를 보강했고, status color token은 객실/콜 상태 의미에만 쓰도록 문서와 validator에 고정했다.

## 잘 된 점

- Epic 5의 핵심 교훈인 current value와 historical snapshot 분리가 Epic 6 전반에 적용됐다.
- Dashboard route/page는 DTO 소비자로 유지됐고, calls/settlements/closing/rooms 계산을 화면 코드에 복제하지 않았다.
- Story 6.3에서 chart library를 추가하지 않고 project-local SVG/CSS primitive로 충분한 그래프와 fallback을 구현했다.
- Story 6.3 review에서 실제 그래프 누락과 negative net sales scaling 문제가 발견되어 Story 6.4 guardrail로 이어졌다.
- Story validator가 단순 파일 존재 확인을 넘어 snapshot fallback 금지, raw error 노출 금지, status color 오용 금지, fake 0값 그래프 금지를 잡도록 진화했다.
- E2E runtime 실패를 통과로 포장하지 않고 `EPERM`, PostgreSQL `ECONNREFUSED`, 대체 검증을 story log에 분리 기록했다.

## 어려웠던 점

- Dashboard는 read-only 표면이지만 upstream source가 `calls`, `settlements`, `closing`, `rooms`로 넓어져 source ownership을 계속 확인해야 했다.
- `마감확정`/`잠금` 월의 매출/상태 집계와 지급/정산 source가 서로 다른 성격을 가진다. 금액 KPI와 payout summary가 섞이지 않도록 `sourceBasis`와 label을 명확히 해야 했다.
- 그래프 요구사항은 “차트처럼 보이는 UI”보다 더 넓었다. 접근 가능한 이름, visible value, legend/label, fallback data, status color 의미 보존까지 검증해야 했다.
- 실제 browser E2E는 반복적으로 환경 제약에 막혔다. 일부는 dev server bind `EPERM`, 일부는 Prisma `Employee` seed 단계의 PostgreSQL `ECONNREFUSED`였다.
- `npx tsc --noEmit`은 오래된 call-ledger/E2E/account-service typing 문제로 계속 실패했다. Epic 6 변경 파일의 type regression 여부를 별도 검증으로 분리해야 했다.

## 핵심 학습

1. Dashboard는 presentation feature가 아니라 source contract feature다. 화면보다 먼저 DTO ownership과 upstream reuse가 고정되어야 한다.
2. Closed/locked month에서 current preview fallback은 편리하지만 위험하다. Missing snapshot은 별도 상태로 드러내야 historical trust가 유지된다.
3. Chart dependency는 필요가 증명되기 전에는 비용이다. Epic 6 범위에서는 local SVG/CSS primitive와 fallback table/list가 더 통제 가능했다.
4. Empty state는 단순 “데이터 없음”이 아니다. no calls, no calculated completed calls, settlement source missing, room status missing, snapshot missing은 운영 의미가 다르다.
5. Static validator는 story guardrail 역할을 할 수 있다. 특히 sandbox에서 runtime E2E가 막힐 때 source contract 회귀를 잡는 보조 안전망이 된다.

## Significant Discoveries

- Chart library는 Epic 6에서 도입하지 않았다. project-local SVG/CSS primitive와 fallback data가 v1 dashboard 요구를 충족했고, dependency 추가는 후속 story에서 다시 근거를 요구해야 한다.
- Snapshot/current source selection은 dashboard에서도 core architecture decision이다. `sourceBasis`가 없는 dashboard extension은 closed/reopened month에서 잘못된 truth를 보여줄 위험이 크다.
- Browser E2E blocker는 일시적 테스트 실패가 아니라 검증 운영 리스크다. Epic 7의 계산 대조는 DB 연결 가능성과 seed 안정성을 별도 확인해야 한다.

## Readiness Assessment

- Testing and quality: Story 6.1~6.4 unit tests, static validators, lint chain, Playwright `--list`, Story 6.4 source guardrail E2E가 기록되어 있다. Full browser E2E는 환경 제약으로 일부 또는 전체가 중단됐다.
- Deployment: repository 내 production 배포 완료 증거는 확인하지 않았다.
- Stakeholder acceptance: 별도 명시 문서는 확인하지 않았다.
- Technical health: dashboard service boundary, sourceBasis, route access, local chart primitive, status color/empty/error guardrail은 안정적이다.
- Unresolved blockers: DB connectivity 기반 full E2E, pre-existing full TypeScript check failures, Story 4.1 만근 source gap이 남아 있다.
