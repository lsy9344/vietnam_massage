# Story 4.3: 귀케어 근무상태 입력

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 정산 담당,
I want 귀케어사별 일일 근무상태를 입력하기를,
so that 귀케어 일일정산에서 정상 근무자만 지급 대상으로 삼을 수 있다.

## Acceptance Criteria

1. 정산 담당자가 귀케어 일정산 화면에 접근해 운영월과 날짜를 선택하면, active 귀케어사 4명의 일별 근무상태 입력 목록이 표시되고 귀케어사 식별은 이름 문자열이 아니라 `Employee.id` 기준으로 처리된다.
2. 각 귀케어사 행의 상태 선택 목록은 active `CodeItem` 중 `codeType === "ATTENDANCE_STATUS"`인 코드만 사용하며 기본 업무값 `정상`, `휴무`, `지각`, `조퇴`, `결근`을 지원한다. 저장 payload는 표시명이 아니라 stable code 값을 보낸다.
3. 귀케어사 근무상태가 `정상` 또는 stable code `NORMAL`이면 지급 대상이고, 후속 Story 4.4 귀케어 일일정산의 균등 분배 대상 목록으로 재사용할 수 있다.
4. 귀케어사 근무상태가 `휴무`, `지각`, `조퇴`, `결근` 또는 동등 stable code이면 지급 대상이 아니며, 화면과 DTO에는 제외 사유가 해당 근무상태로 표시된다.
5. 운영월 상태가 `잠금`이면 서버에서 귀케어 근무상태 생성/수정을 차단하고, UI는 잠긴 운영월 안내와 disabled 입력 상태를 제공한다.
6. 귀케어 근무상태 생성 또는 수정이 저장되면 행위자 계정, 대상 귀케어사 `Employee.id`, 운영월, 근무일자, 변경 전후 상태, 변경 시각이 감사 로그로 기록되고 지급액에 영향을 줄 수 있는 변경으로 분류된다.
7. 운영월 또는 날짜가 변경되거나 저장이 진행 중일 때 명확한 로딩/저장 상태를 표시하고, 이전 날짜의 근무상태가 새 날짜의 값처럼 보이지 않는다.
8. 4명 목록, 상태 선택, 정상 지급 대상 판정, 비정상 제외, 운영월 잠금 차단, 감사 로그, 후속 귀케어 정산 재사용 DTO를 검증하는 unit/static/e2e 테스트가 통과한다.

## Dev Notes

- 이 story는 귀케어 지급액을 계산하지 않는다. 목표는 후속 Story 4.4가 사용할 “날짜별 정상 근무 귀케어사 목록과 제외 사유”를 신뢰 가능한 저장 데이터로 만드는 것이다. [Source: _bmad-output/planning-artifacts/epics.md#Story-4.3]
- Epic 4는 일정산과 운영팀 인센 epic이다. 4.3은 4.4 귀케어 일일정산의 선행 데이터이며, 4.4는 방문완료 콜의 귀케어 풀을 정상 근무 귀케어사에게 균등 분배한다. [Source: _bmad-output/planning-artifacts/epics.md#Epic-4]
- 귀케어팀 기본 인원은 4명이고 stable group code는 `EARCARE`다. 직원 stable reference는 `Employee.id`와 `staffCode`이며 displayName을 downstream 저장 키로 쓰면 안 된다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 근무상태 기본 코드는 `정상`, `휴무`, `지각`, `조퇴`, `결근`이며 `CodeItem`의 `ATTENDANCE_STATUS`가 소유한다. 빈 값이나 임의 문자열을 저장하지 않는다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 운영월은 `OperatingMonth`가 소유하고 상태값은 `작성중`, `검토중`, `마감확정`, `잠금`이다. 잠금 상태는 지급 영향 데이터 수정을 서버에서 차단해야 한다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 지급액에 영향을 주는 출퇴근/근무상태 변경은 감사 로그 대상이다. `recordAuditEvent()`는 JSON 직렬화 가능한 값만 받으므로 `Date` 객체를 snapshot에 직접 넣지 않는다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 현재 `prisma/schema.prisma`에는 `Employee`, `OperatingMonth`, `CodeItem`, `AuditLog`가 있지만 귀케어 근무상태 저장 모델은 없다. 4.3 구현은 Prisma schema와 migration을 포함해야 한다. [Source: prisma/schema.prisma]
- 현재 `/settlements`는 Story 4.2 마사지사 일일정산 화면으로 구현되어 있다. 이 파일의 route access, 운영월/날짜 selector, dense table 스타일을 보존하고, 귀케어 화면은 `/settlements/earcare` subroute로 추가하는 것이 기존 4.2 회귀를 줄인다. [Source: src/app/(erp)/settlements/page.tsx]
- `src/modules/settlements/therapist-daily-settlement-service.ts`는 read-only 계산 service이며 귀케어 근무상태 mutation과 섞지 않는다. 다만 service 구조, Zod 검증, injectable prismaClient test pattern은 재사용한다. [Source: src/modules/settlements/therapist-daily-settlement-service.ts]
- `src/modules/settlements/README.md`는 이미 earcare attendance를 settlements module 책임으로 명시한다. 4.3 완료 시 새 service/API 요약과 downstream 4.4 재사용 규칙을 갱신한다. [Source: src/modules/settlements/README.md]
- `requireRouteAccess()`는 route 접근을 redirect로 처리하고, `requirePermission("payout:write")`는 administrator/settlement_manager에게만 mutation을 허용한다. 화면 접근만 믿지 말고 Server Action에서 권한을 재조회한다. [Source: src/lib/authorization.ts]
- Server Actions는 서버에서 실행되는 form mutation 경계이며, 공식 Next.js 문서도 action 내부에서 인증/인가를 검증하라고 명시한다. 저장 후 화면 갱신은 `revalidatePath()`를 사용한다. [Source: https://nextjs.org/docs/app/guides/forms; Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
- Prisma transaction은 여러 read/write가 하나의 성공/실패 단위여야 할 때 사용한다. 4.3 저장과 감사 로그 기록은 같이 성공하거나 같이 실패해야 한다. [Source: https://www.prisma.io/docs/orm/prisma-client/queries/transactions]
- 입력 검증은 프로젝트의 기존 Zod 패턴을 따른다. `.safeParse()`는 성공/실패 result union을 반환하므로 action field error mapping에 적합하다. [Source: https://zod.dev/basics]
