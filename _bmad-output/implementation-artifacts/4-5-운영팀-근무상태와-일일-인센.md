# Story 4.5: 운영팀 근무상태와 일일 인센

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 정산 담당,
I want 운영팀의 일별 근무상태와 일 총콜 기준 일일 인센을 계산하기를,
so that 정상 근무한 운영팀 직원에게만 원본 기준의 일일 인센을 지급할 수 있다.

## Acceptance Criteria

1. 정산 담당자가 운영팀 근무/인센 화면에서 운영월과 날짜를 선택해 조회하면 팀장, 카운터1, 카운터2, 웨이터1, 웨이터2 대상자의 일별 근무상태가 표시된다. 직원 식별과 저장 키는 표시명이 아니라 `Employee.id`다.
2. 운영팀 근무상태 선택 목록은 active `CodeItem` 중 `codeType === "ATTENDANCE_STATUS"`를 사용하며 `정상`, `휴무`, `지각`, `조퇴`, `결근`을 지원한다. 저장 값은 stable `statusCode`다.
3. 운영팀 근무상태 생성/수정은 `payout:write` 권한, 운영월 날짜 범위 검증, `잠금` 운영월 서버 차단을 통과해야 한다. 저장 성공 후 화면은 같은 route를 revalidate해 근무상태와 일일 인센 계산을 최신 데이터로 다시 보여준다.
4. 운영팀 근무상태 생성/수정은 행위자, 대상 운영팀 직원, 운영월, 날짜, 변경 전후 상태, 변경 시각, `payoutImpact: true`를 plain JSON 감사 로그로 기록한다. audit event action은 `ops_attendance.created` / `ops_attendance.changed`, reason은 `payout_affecting`을 사용한다.
5. 일 총콜은 calls domain의 완료 콜 계산 결과 중 `calculationStatus === "calculated"`인 `방문완료`/`VISIT_COMPLETE` 콜의 `opsCallCredit` 합계다. `예약`, `사용중`, `청소중`, `노쇼`, `취소`, `course_policy_missing`, `therapist_rate_missing`, `second_therapist_required` row는 인센 기준 콜 수에 포함하지 않고 warning/제외 근거로만 표시한다.
6. 적용월에 활성 운영팀 일일 인센 정책이 있으면 가장 높은 충족 threshold를 적용한다. 30 이상 40 미만은 개인별 50,000 VND, 40 이상 50 미만은 100,000 VND, 50 이상은 200,000 VND다. 30 미만이면 개인별 0 VND와 `30콜 미만` 근거를 표시한다.
7. `NORMAL` 또는 표시명 `정상`인 운영팀 직원만 지급 대상이다. 정상 상태가 아닌 운영팀 직원은 지급액 0 VND이며 제외 사유는 근무상태 표시명으로 표시한다.
8. 적용월에 `OpsDailyIncentiveRule`이 없으면 조용히 0으로 가정하지 않는다. 결과 DTO는 `ruleStatus: "missing_policy"`와 한국어 warning을 반환하고 화면에는 정책 없음/미확정 근거를 표시한다.
9. 운영월이 `잠금` 상태여도 운영팀 근무상태와 일일 인센 계산 결과는 읽기 전용으로 조회할 수 있다. 근무상태 select/save는 disabled이며 서버 action도 차단한다.
10. 이 story는 일일 인센 계산 결과를 저장하지 않는다. 인센 지급액 직접 수정 UI/action, payout result table, 월마감 snapshot, `/closing` 변경, 운영팀 월 인센 미리보기는 만들지 않는다.
11. 대상자 5명, 근무상태 선택, 잠금 차단, 감사 로그, 방문완료 기준 일 총콜, `opsCallCredit` 합산, 30/40/50 threshold, 30 미만 0원, 정상 직원만 지급, 정책 없음 warning, 권한 경계를 검증하는 unit/static/e2e 테스트가 통과한다.

## Dev Notes

- Epic 4는 일정산과 운영팀 인센 epic이다. Story 4.5는 운영팀 5명의 일별 근무상태를 입력하고, 방문완료 기준 일 총콜으로 정상 근무 직원의 개인별 일일 인센을 계산한다. [Source: _bmad-output/planning-artifacts/epics.md#Story-4.5]
- PRD FR-24는 운영팀 대상자가 팀장, 카운터1, 카운터2, 웨이터1, 웨이터2이며 일 총콜 30/40/50 이상 기준 개인별 50,000/100,000/200,000 VND를 지급한다고 정의한다. [Source: _bmad-output/planning-artifacts/prds/prd-vietnam_massage-2026-06-07/prd.md#FR-24]
- 운영팀 일일 인센 정책은 Story 1.8의 `OpsDailyIncentiveRule` 모델과 적용월 이력이 source of truth다. 정책이 없을 때 downstream 계산은 조용히 0으로 가정하지 말고 명시적 not-found/warning을 처리한다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- 운영팀 기본 직원 seed는 팀장 `OPS-LEAD-001`, 카운터 `OPS-COUNTER-DAY-001`/`OPS-COUNTER-NIGHT-001`, 웨이터 `OPS-WAITER-DAY-001`/`OPS-WAITER-NIGHT-001` 5명이다. service는 표시명 하드코딩 대신 active `Employee.employeeGroup === "OPERATIONS"`와 stable `Employee.id`를 사용하되, E2E는 shared DB의 다른 운영팀 계정 fixture가 결과를 오염시키지 않도록 story 전용 staffCode/운영월을 격리한다. [Source: src/modules/masters/employee-schema.ts]
- 콜 원장은 정산 source of truth다. `방문완료`가 아닌 상태는 매출, 수당, 귀케어 풀, 콜인정에 포함하지 않는다. 운영팀 일 총콜도 이 원칙을 따른다. [Source: _bmad-output/project-context.md#절대-놓치면-안-되는-규칙]
- Story 2.3 기준 완료 콜 계산은 calls domain service가 소유한다. `listCompletedServiceCallCalculationsForDate()`는 `opsCallCredit`를 이미 반환하므로 운영팀 인센 service는 이를 합산하고 UI에서 콜 상태/코스 정책을 재계산하지 않는다. [Source: src/modules/calls/service-call-service.ts]
- 지급액에 영향을 주는 근무상태 변경은 감사 로그 대상이다. 감사 로그 snapshot은 plain JSON이어야 하며 Date 객체는 ISO 문자열로 넘긴다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- Next.js 공식 Forms guide는 Server Action 내부에서도 인증/인가를 검증해야 한다고 안내한다. 운영팀 근무상태 저장 action은 `/settlements/operations` 화면 접근과 별개로 `requirePermission("payout:write")`를 호출해야 한다. [Source: https://nextjs.org/docs/app/guides/forms]
- `revalidatePath()`는 서버 환경에서 특정 path의 cached data를 on-demand invalidation한다. 저장 후 같은 route의 근무상태와 일일 인센 계산을 최신 상태로 보여주려면 `revalidatePath("/settlements/operations")`를 사용한다. [Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
- Prisma transaction은 read/write 연산 묶음을 전체 성공 또는 전체 실패로 처리한다. attendance upsert와 audit log create는 하나의 transaction 안에서 실행해 audit 실패 시 attendance write도 rollback되게 한다. [Source: https://www.prisma.io/docs/orm/prisma-client/queries/transactions]
- Zod `safeParse()`는 성공/실패 result union을 반환한다. 기존 service/action처럼 field/domain error mapping에 이 패턴을 사용한다. [Source: https://zod.dev/basics]
- 현재 package baseline은 Next.js `16.2.7`, React `19.2.7`, Prisma `7.8.0`, Zod `4.1.12`, Playwright `1.60.0`, Node test + tsx다. 새 dependency를 추가하지 않는다. [Source: package.json]
