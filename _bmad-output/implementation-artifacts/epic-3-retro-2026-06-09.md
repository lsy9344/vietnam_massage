## 완료 결과

### Story 3.1: 객실 상태 DTO와 최신 활성 콜 계산

`RoomStatusDto`와 `listRoomStatuses()`를 도입해 `/live`, `/rooms`, `/tv`가 공유할 조회 전용 계약을 만들었다. 활성 상태는 `예약`/`RESERVED`, `사용중`/`IN_USE`/`USING`, `청소중`/`CLEANING`으로 제한했고, `방문완료`/`VISIT_COMPLETE`, `노쇼`/`NO_SHOW`, `취소`/`CANCELED`는 객실 점유 상태에서 제외했다.

### Story 3.2: 첫 화면 실시간 객실/콜 현황

`/live`를 읽기 전용 운영 현황 화면으로 구현했다. 객실 카드는 `RoomStatusDto`를 직접 소비하고, 콜 상태/KPI 요약은 `getDailyCallLedgerSummary()`를 재사용해 UI 계산 재구현을 피했다.

### Story 3.3: 객실 현황 화면과 웨이터 안내 문구

`/rooms`를 웨이터와 조회 전용 사용자의 객실 현황 landing으로 구현했다. 안내 문구 source of truth를 `ROOM_STATUS_GUIDANCE_TEXT`로 고정했고, route-local 문구 복제를 피했다.

### Story 3.4: TV 현황판 fullscreen route

`/tv`를 ERP shell 밖의 fullscreen 읽기 전용 route로 분리했다. `RoomStatusCard variant="tv"`와 `RoomStatusRefreshController variant="tv"`를 재사용해 TV 전용 표시와 shared refresh 계약을 동시에 충족했다.

### Story 3.5: 상태 badge, 종료확인 주의 표현, 접근성 polish

`StatusBadge`, `RoomStatusCard`, `globals.css`를 status presentation source of truth로 잠갔다. 모든 상태는 색상, 텍스트, 글리프를 함께 표시하고, `종료확인` animation은 text-bearing content가 아니라 card ring pseudo-element에만 적용되도록 수정했다.

## 잘 된 점

- 도메인 계산을 먼저 고정한 뒤 화면을 얹은 순서가 효과적이었다. Story 3.1의 `RoomStatusDto`가 이후 `/live`, `/rooms`, `/tv` 구현의 흔들림을 줄였다.
- `/live`와 `/rooms`와 `/tv`가 같은 `RoomStatusCard`와 `StatusBadge`를 쓰게 되어 상태 색상/문구/글리프 drift가 크게 줄었다.
- validator를 story별 완료 조건으로 둔 덕분에 docs, project context, E2E, source file sync 누락을 빠르게 잡을 수 있었다.
- TV fullscreen 요구를 `(erp)` route 안에서 억지로 해결하지 않고 `src/app/tv/page.tsx`로 분리한 결정이 구조적으로 맞았다.
- 리뷰 auto-fix가 실제 결함을 잡았다. 예: 잘못된 `serviceDate` rollover, 불가능한 `startTime`, TV stale text contrast, nested attention animation.

## 어려웠던 점

- 초반 rooms 문서는 `RoomStatusView`/`RoomStatusSnapshot` 같은 모호한 이름과 `RoomStatusDto` 사이에 용어 drift 위험이 있었다.
- live/rooms/tv 화면마다 상태 계산을 다시 만들 유혹이 있었고, 이를 막기 위해 문서와 validator에 "UI 계산 재구현 금지"를 반복해서 박아야 했다.
- Playwright web server 실행은 현재 sandbox에서 `127.0.0.1:3000` listen `EPERM`으로 막히는 반복 패턴이 있었다. 각 story는 이 제약을 명시하고 가능한 대체 검증을 기록했다.
- 접근성 polish에서 `종료확인` attention을 badge와 card에 중첩 적용하면서 텍스트 대비를 낮출 수 있는 문제가 리뷰에서 발견됐다.

## 핵심 학습

1. 실시간 상태 표면은 transport보다 source-of-truth가 먼저다. polling, refresh controller, fullscreen route는 바뀔 수 있지만 `listRoomStatuses()`와 `RoomStatusDto`는 흔들리면 안 된다.
2. "색상만으로 상태 전달 금지"는 UI 취향 문제가 아니라 운영 안전 규칙이다. 상태 label과 glyph를 강제한 것은 TV 거리 가독성과 접근성 모두에 도움이 됐다.
3. 문서 업데이트는 story 완료의 부가 작업이 아니라 회귀 방지 장치다. Epic 3에서는 docs/project-context/validator 삼각형이 drift를 줄였다.
4. 다음 Epic의 정산 계산도 같은 원칙이 필요하다. UI에서 계산하지 않고 domain service가 산출 근거와 함께 결과를 소유해야 한다.

## Significant Discoveries

Epic 4의 전체 방향을 바꿀 정도의 중대한 제품 범위 변경은 발견되지 않았다. 다만 Epic 4 실행 방식에는 강한 제약이 생겼다: 정산 계산은 반드시 domain service에서 수행하고 UI는 산출 근거 DTO를 렌더링해야 한다.

## Readiness Assessment

- Testing and quality: Epic 3 validators와 unit/domain tests는 통과 기록이 있다. 일부 Playwright E2E는 sandbox listen 제약으로 차단되었고, story별로 차단 조건과 대체 검증이 기록됐다.
- Deployment: repository 내 배포 상태 증거는 확인하지 않았다.
- Stakeholder acceptance: 별도 명시 문서는 확인하지 않았다.
- Technical health: 공유 DTO/service/card/badge 구조가 안정적이다. 문서 drift는 본 retrospective 후 별도 audit에서 검증 후 필요한 항목만 수정한다.
- Unresolved blockers: Epic 4를 막는 Epic 3 기능 blocker는 발견되지 않았다.

## Commitments

- Epic 4에서는 UI 계산 재구현 금지 원칙을 settlement 화면에도 적용한다.
- 새 domain service가 생기면 docs/project-context/story validator를 같이 갱신한다.
- 접근성/상태 표시 규칙은 dashboard와 settlement evidence UI에도 그대로 확장한다.
- sandbox에서 실행 불가능한 검증은 통과로 쓰지 않고 차단 조건과 대체 검증을 분리 기록한다.
