# Story 5.3: 월마감 검토와 확정 스냅샷

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 정산 담당,
I want 월마감을 검토 후 확정하고 확정 시점의 계산 근거를 스냅샷으로 저장하기를,
so that 이후 정책이나 원장 변경으로 과거 지급액이 흔들리지 않게 할 수 있다.

## Acceptance Criteria

1. 운영월 상태가 `작성중`이면 administrator 또는 settlement_manager가 `/closing`에서 `검토 시작`을 실행해 운영월 상태를 `검토중`으로 변경할 수 있고, 상태 변경은 `operating_month.status_changed` 감사 로그에 actor, target operating month, before/after state, timestamp를 남긴다.
2. `/closing` 화면은 `작성중 -> 검토중 -> 마감확정 -> 잠금` 수평 스테퍼를 표시하고, 현재 상태와 사용자 역할 기준으로 가능한 action만 표시한다. 권한 없는 route 접근은 기존 `requireRouteAccess("/closing")`로 유지한다.
3. 운영월 상태가 `검토중`이면 administrator 또는 settlement_manager가 서버 action을 통해 마감 확정을 요청할 수 있고, counter/waiter/read_only_viewer 또는 비활성/잠금 계정은 서버에서 차단된다.
4. 확정 처리는 `closing` domain service가 소유한다. UI component는 스냅샷 생성, 지급액 계산, 상태 전이, 감사 로그 payload를 직접 계산하지 않는다.
5. 확정 시점에는 현재 `listMonthlyClosingPreview()` 결과를 기반으로 콜 원장 집계, 적용 정책/경고/evidence, 마사지사/운영팀/귀케어 지급 row, totals, 계산 근거를 persistence snapshot으로 저장한다.
6. 스냅샷은 mutable 표시명/정책 변경에 의존하지 않도록 `Employee.id`, `staffCode`, 표시명, `Course.code`, 금액 number, ISO 날짜, warning/evidence 문자열, totals를 plain JSON 또는 명시 컬럼으로 보존한다. `Date` 객체, class instance, `NaN`, formatted money string만 있는 값은 저장하지 않는다.
7. 스냅샷 저장과 운영월 상태 `마감확정` 변경, `monthly_close.confirmed` 감사 로그 기록은 하나의 atomic transaction으로 처리된다. 일부만 성공한 상태가 남으면 안 된다.
8. 확정 완료 후 운영월 상태는 `마감확정`이고, 확정 시각, 확정자 account id, snapshot id가 조회 DTO와 UI에 표시된다.
9. 이미 `마감확정` 또는 `잠금`인 운영월에 다시 확정을 요청하면 중복 확정은 차단되고, 사용자에게 `현재 상태에서는 마감 확정할 수 없습니다.` 또는 동등한 한국어 오류가 표시된다. 새 snapshot/audit/status update가 생성되면 안 된다.
10. 확정 이후 마스터 정책, 직원명, 콜 원장이 변경되어도 확정 스냅샷 조회 값은 자동 재계산되지 않는다. `/closing`은 현재 기준 미리보기와 확정 스냅샷을 명확히 구분해 표시한다.
11. `monthly_close.confirmed` 감사 로그는 actor, target operating month, before/after state, snapshot id, timestamp를 포함하고 dot notation action 규칙을 통과한다.
12. unit/static/e2e 테스트는 작성중->검토중, 검토중->마감확정, 권한 차단, 중복 확정 차단, 스냅샷 불변성, audit payload, transaction 실패 시 부분 반영 없음, 현재 미리보기/확정 스냅샷 구분을 검증한다.

## Dev Notes

- Epic 5의 목표는 월 지급액을 미리보기, 검토, 확정, 잠금, 재오픈 흐름으로 처리하고 확정 스냅샷을 보존하는 것이다. Story 5.3은 처음으로 월마감 persistence, 상태 전이, 감사 로그를 도입하는 story다. [Source: `_bmad-output/planning-artifacts/epics.md` Epic 5, Story 5.3]
- FR29는 월마감을 미리보기, 검토, 확정, 잠금 단계로 처리하고 확정 시 마감 스냅샷을 저장해야 한다. NFR6은 확정 스냅샷이 이후 설정 변경으로 자동 재계산되어 흔들리면 안 된다고 요구한다. [Source: `_bmad-output/planning-artifacts/epics.md` FR29, NFR6]

- Internal UI mutation은 Next.js App Router Server Actions를 기본으로 사용하고, Server Action은 인증/권한/검증/domain service 호출/ActionResult mapping만 담당한다. broad REST API를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md` API & Communication Patterns]

- 날짜는 ISO `YYYY-MM-DD`, date-time은 ISO string, 금액은 integer VND number다. UI formatting 결과만 snapshot source of truth로 저장하지 않는다.
