# Story 5.5: 관리자 사유 기반 재오픈

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 관리자,
I want 잠긴 월마감을 사유 입력 후 재오픈하기를,
so that 잘못 확정된 운영월을 통제된 절차로 수정 가능 상태로 되돌릴 수 있다.

## Acceptance Criteria

1. 운영월 상태가 `잠금`일 때 administrator만 `/closing`에서 재오픈을 실행할 수 있다. `settlement_manager`의 기존 `closing:write` 권한만으로는 재오픈할 수 없다.
2. 재오픈 입력에는 사유가 필수다. trim 후 빈 문자열이거나 정책상 너무 짧은 값이면 재오픈은 차단되고 운영월, 스냅샷, 감사 로그는 변경되지 않는다.
3. 유효한 사유로 재오픈하면 운영월 상태는 `검토중`으로 변경된다. 이 story의 domain rule은 `잠금 -> 검토중`이다. `검토중`은 Story 5.4의 payout lock guard 대상이 아니므로 지급 영향 데이터 수정이 다시 가능하다.
4. 재오픈 처리는 `closing` domain service가 소유한다. UI component나 Server Action은 상태 전이, snapshot versioning, 감사 로그 payload를 직접 구현하지 않는다.
5. 재오픈은 `잠금` 상태에서만 허용된다. `작성중`, `검토중`, `마감확정` 상태에서 요청하면 현재 상태에서 재오픈할 수 없다는 한국어 오류와 domain code를 반환하고 부작용을 만들지 않는다.
6. 재오픈 완료 시 `monthly_close.reopened` 감사 로그가 기록된다. `targetType`은 `monthly_close`, `targetId`는 재오픈 대상인 최신 확정 스냅샷 id이며, before/after에는 operating month id/monthKey, before status `잠금`, after status `검토중`, reason, reopenedAt, reopenedByAccountId, snapshot id가 plain JSON으로 포함된다.
7. 기존 확정 스냅샷은 삭제하거나 overwrite하지 않는다. 재오픈 이력과 이전 스냅샷은 감사/조회 목적으로 구분 가능해야 한다.
8. 재오픈 후 다시 `confirmMonthlyClose()`를 실행하면 새 `MonthlyClosing` snapshot row가 생성되고 이전 snapshot과 새 snapshot은 version 또는 sequence로 구분 가능해야 한다.
9. `getMonthlyClosingSnapshot()` 또는 후속 snapshot 조회 API는 versioned snapshot 구조에서도 기존 호출자가 최신 스냅샷을 읽을 수 있게 유지한다. 재오픈 상태에서는 이전 확정 스냅샷을 `이전 확정 스냅샷` 또는 동등한 문구로 현재 미리보기와 구분한다.
10. `/closing` UI는 `잠금` 상태에서 재오픈 affordance를 보여준다. administrator는 사유 입력 후 실행할 수 있고, 비관리자는 재오픈 버튼/폼을 보되 비활성 및 권한 없음 이유를 확인할 수 있다.
11. 재오픈 성공 후 `/closing`, 지급 영향 화면, 월간 지급/KPI 성격의 조회 화면은 revalidation되어 `검토중` 상태와 수정 가능 상태를 반영한다. 서버 guard가 source of truth다.
12. unit/static/e2e 테스트는 관리자 재오픈, 사유 필수, 비관리자 차단, 비잠금 상태 차단, `monthly_close.reopened` 감사 로그 reason 기록, 기존 스냅샷 보존, 재확정 시 새 스냅샷 생성 및 구분, 재오픈 후 지급 영향 write 허용을 검증한다.

### Architecture Compliance

- `closing` domain service가 상태 전이, snapshot versioning, audit payload를 소유한다.
- Server Action은 authenticate/authorize/validate/call/revalidate만 수행한다.
- 재오픈 권한은 `closing:write`와 분리한다. settlement manager가 월마감 확정/잠금은 할 수 있어도 재오픈은 할 수 없다.
- 재오픈은 snapshot 삭제가 아니다. 이전 snapshot은 historical record이고, 후속 재확정은 새 version snapshot이다.
- `monthly_close.reopened`는 후속 지급 영향 변경 이력을 대체하지 않는다. 재오픈 후 콜/정산/정책 변경은 각 domain audit event로 따로 남는다.
- Audit snapshot은 JSON-safe여야 한다. reason은 plain string, date/time은 ISO string, 금액은 number, 객체는 plain object만 사용한다.
- UI는 read-only/disabled 힌트를 제공하지만 permission과 state transition source of truth는 server/domain boundary다.
