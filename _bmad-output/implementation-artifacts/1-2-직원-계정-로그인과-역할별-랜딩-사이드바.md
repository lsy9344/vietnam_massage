# Story 1.2: 직원 계정 로그인과 역할별 랜딩/사이드바

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 직원 사용자,
I want 내 계정으로 로그인하면 역할에 맞는 첫 화면과 메뉴만 보이기를,
so that 각 사용자 역할이 허용된 ERP 업무만 안전하게 수행할 수 있다.

## Acceptance Criteria

1. **직원 계정 로그인**
   - **Given** 관리자에 의해 생성된 직원 계정이 있다
   - **When** 직원이 이메일 또는 계정 ID와 비밀번호로 로그인한다
   - **Then** 인증은 Auth.js/NextAuth 기반으로 처리된다
   - **And** public signup은 제공되지 않는다.

2. **Argon2id 비밀번호 저장**
   - **Given** 직원 계정의 비밀번호가 저장된다
   - **When** 계정이 생성되거나 비밀번호가 변경된다
   - **Then** 비밀번호는 Argon2id hash로 저장된다
   - **And** plaintext 또는 reversible password material은 저장되지 않는다.

3. **역할별 landing**
   - **Given** 로그인 사용자가 관리자, 카운터, 웨이터, 정산 담당, 조회 전용 중 하나의 역할을 가진다
   - **When** 로그인 후 ERP에 진입한다
   - **Then** 사용자는 역할별 landing 화면으로 이동한다
   - **And** 관리자/주인은 첫화면 실시간 현황, 카운터는 콜/예약 입력 원장, 정산 담당은 정산 화면, 웨이터와 조회 전용은 객실 현황으로 이동한다.

4. **역할별 sidebar 가시성**
   - **Given** 사용자가 sidebar를 본다
   - **When** 사용자의 역할에 허용되지 않은 도메인 그룹이 있다
   - **Then** 해당 그룹은 비활성 표시가 아니라 숨겨진다
   - **And** 허용된 그룹은 운영 현황, 콜 원장, 정산, 월마감, 대시보드, 마스터 설정, 감사 로그의 고정 순서를 유지한다.

5. **직접 route 접근 차단**
   - **Given** 사용자가 권한이 없는 route에 직접 접근한다
   - **When** 서버에서 요청을 처리한다
   - **Then** route/layout 또는 server boundary에서 접근을 차단한다
   - **And** UI에서 메뉴를 숨기는 것만으로 보안을 대신하지 않는다.

6. **민감 action 서버 재검사**
   - **Given** 사용자가 지급액, 수당, 마감, 직원 정보에 영향을 주는 작업을 요청한다
   - **When** Server Action 또는 Route Handler가 실행된다
   - **Then** 현재 계정 상태와 권한을 서버에서 다시 확인한다
   - **And** session payload의 role hint만 신뢰하지 않는다.

7. **비활성/잠금 계정 차단**
   - **Given** 직원 계정이 비활성 상태이거나 반복 로그인 실패로 잠겼다
   - **When** 해당 계정으로 로그인하려 한다
   - **Then** 로그인은 거부된다
   - **And** 사용자에게 안전한 한국어 오류 메시지가 표시된다.

8. **검증**
   - **Given** 개발자가 인증/권한 기능을 검증한다
   - **When** 역할별 route 접근과 sidebar 노출을 테스트한다
   - **Then** 각 역할은 허용된 landing과 메뉴만 볼 수 있다
   - **And** 권한 없는 민감 action은 서버에서 실패한다.

### Epic 1 Context

Epic 1 builds ERP operating foundations and access control before payout-affecting workflows. Story 1.2 is the security gate before Story 1.3 audit logging and before operating month, room, code, employee, course, rate, and incentive masters. Do not defer server-side authorization to later stories; later stories must reuse the guard created here. [Source: `_bmad-output/planning-artifacts/epics.md` - Epic 1, Story 1.2]

### Story 1.2 Specific Requirements

- Authentication must use Auth.js/NextAuth, and public signup must not exist. Staff accounts are provisioned by an administrator or server-only seed/service. [Source: `_bmad-output/planning-artifacts/epics.md` - Story 1.2 AC; `_bmad-output/planning-artifacts/architecture.md` - Authentication & Security]
- Passwords must be Argon2id hashes only. No plaintext password, reversible material, or "temporary password" storage in DB, seed output, logs, fixtures, or comments. [Source: `_bmad-output/planning-artifacts/epics.md` - Additional Requirements; `_bmad-output/planning-artifacts/architecture.md` - Password storage]
- Login identity and employee display data are separate. Use `UserAccount` linked to `Employee`; employee name/role/display changes must not corrupt authentication or historical settlement snapshots. [Source: `_bmad-output/planning-artifacts/epics.md` - Additional Requirements; `_bmad-output/planning-artifacts/architecture.md` - User and employee separation]
- RBAC roles are administrator, counter, waiter, settlement manager, and read-only viewer. Sidebar visibility and landing screens follow role rules, but execution-time authorization must also run in server boundaries. [Source: `_bmad-output/planning-artifacts/epics.md` - Story 1.2 AC; `_bmad-output/planning-artifacts/architecture.md` - Authorization model]
- Sensitive operations affecting calls, payments, discounts, staff assignments, shifts, rate policies, employee data, monthly close state, reopen actions, or audit-visible records must reload current account status and permission from DB. Session role hints are not enough. [Source: `_bmad-output/planning-artifacts/architecture.md` - Execution-time enforcement, Session strategy]

### Architecture Guardrails

- `src/app` owns route composition and layouts. Domain/account logic belongs in `src/modules/*` services or `src/lib/*` helpers, not React components. [Source: `_bmad-output/planning-artifacts/architecture.md` - Project Organization]
- Internal UI mutation should use Server Actions. Route Handlers are limited to Auth.js, health, exports, and future external integrations. Do not create a broad internal REST API. [Source: `_bmad-output/planning-artifacts/architecture.md` - API & Communication Patterns]
- Server Actions and Route Handlers are untrusted entrypoints: authenticate, authorize, validate with Zod, call domain service, then map to `ActionResult<T>`. [Source: `_bmad-output/planning-artifacts/architecture.md` - Domain service boundary, Validation, Action Response Format]
- Prisma model names use PascalCase singular; database tables map to snake_case plural; code fields use camelCase. [Source: `_bmad-output/planning-artifacts/architecture.md` - Naming Patterns]
- Environment secrets must stay server-only. Do not commit `.env*`; update `.env.example` only. [Source: `_bmad-output/planning-artifacts/architecture.md` - Environment variables]
- Static export remains forbidden because authentication, Server Actions, Route Handlers, and Prisma require Node runtime. [Source: `_bmad-output/planning-artifacts/architecture.md` - Infrastructure & Deployment]
