# Story 4.4: 귀케어 일일정산

Status: done

<!-- Note: Validation completed during create-story. -->

## Story

As a 정산 담당,
I want 방문완료 콜의 귀케어 풀을 정상 근무 귀케어사에게 균등 분배하기를,
so that 귀케어 지급액을 원본 엑셀 규칙과 같은 기준으로 계산할 수 있다.

## Acceptance Criteria

1. 정산 담당자가 `/settlements/earcare`의 귀케어 일정산 화면에서 운영월과 날짜를 선택해 조회하면, 해당 날짜의 귀케어 풀 합계, 정상 근무 지급 대상 수, 귀케어사별 지급액, 산출 근거가 함께 표시된다.
2. 당일 귀케어 풀 합계는 `calculationStatus === "calculated"`인 `방문완료`/`VISIT_COMPLETE` 서비스 콜의 `earcarePoolAmount` 합계다. `예약`, `사용중`, `청소중`, `노쇼`, `취소` 및 `course_policy_missing`, `therapist_rate_missing`, `second_therapist_required` row는 지급 풀에 포함하지 않고 warning/제외 근거로만 표시한다.
3. 해당 날짜에 `NORMAL` 또는 표시명 `정상`인 귀케어사가 1명 이상 있으면 귀케어 풀을 정상 근무 지급 대상에게 균등 분배한다. 정상 상태가 아닌 귀케어사는 지급액 0원이며 제외 사유는 Story 4.3의 근무상태 표시명을 사용한다.
4. 해당 날짜에 정상 근무 귀케어사가 0명이면 모든 귀케어사 개인 지급액은 0원이고, `정상 근무자 0명`과 미분배 귀케어 풀 금액을 산출 근거로 표시한다.
5. 금액은 whole-number VND로 계산한다. 나누어떨어지지 않는 금액은 domain service에서 `baseShare = Math.floor(poolTotal / eligibleCount)`로 계산하고, 남은 `remainderAmount`는 `sortOrder`, `staffCode`, `Employee.id` 순으로 앞선 정상 근무자에게 1 VND씩 배정해 총 지급액 합계가 귀케어 풀 합계와 일치하게 한다. 이 반올림/잔여 배분 규칙은 DTO와 테스트에 드러난다.
6. 귀케어 근무상태가 변경된 뒤 다시 조회하면 지급 대상, 제외 사유, 개인 지급액, 잔여 배분 근거가 최신 근무상태 기준으로 갱신된다. 근무상태 변경 감사 로그는 Story 4.3의 `earcare_attendance.*` 이벤트가 소유한다.
7. 운영월이 `잠금` 상태여도 귀케어 지급액 계산 결과는 읽기 전용으로 조회할 수 있다. 지급액 직접 수정 UI, 지급액 저장 action, 정산 snapshot table, 월마감 확정 데이터 변경은 만들지 않는다.
8. 방문완료 기준 풀 합계, 비완료/미계산 row 제외, 정상 근무자 균등 분배, 정상근무자 0명 지급액 0원, 근무상태 변경 반영, 반올림/잔여 배분 규칙, 읽기 전용 경계, 권한 경계를 검증하는 unit/static/e2e 테스트가 통과한다.

## Dev Notes

- Epic 4는 일정산과 운영팀 인센 epic이다. Story 4.4는 Story 4.3의 귀케어 근무상태를 사용해 방문완료 콜 귀케어 풀을 정상 근무 귀케어사에게 분배한다. [Source: _bmad-output/planning-artifacts/epics.md#Story-4.4]
- PRD FR-23은 `방문완료` 콜의 귀케어 풀을 해당일 정상 근무 귀케어사에게 균등 분배하는 기능이다. FR-22는 정상 상태만 지급 대상이라고 정의한다. [Source: _bmad-output/planning-artifacts/prds/prd-vietnam_massage-2026-06-07/prd.md#FR-23]
- 콜 원장은 정산의 source of truth다. `방문완료`가 아닌 상태는 매출, 수당, 귀케어 풀, 콜인정에 포함하지 않는다. [Source: _bmad-output/project-context.md#절대-놓치면-안-되는-규칙]
- Story 2.3/2.5 기준 completed-call 계산은 `calls` domain service가 소유한다. `calculationStatus === "calculated"`인 완료 콜만 금액/귀케어풀 집계에 포함한다. [Source: _bmad-output/project-context.md#기술-스택-및-버전]
- `listCompletedServiceCallCalculationsForDate()`는 후속 정산/대시보드가 사용할 완료 콜 계산 결과를 반환한다. 이 helper가 반환하는 `earcarePoolAmount`를 지급 풀 원천으로 사용하고, UI에서 `CoursePolicy.earcarePoolAmount`를 직접 조회하거나 계산하지 않는다. [Source: src/modules/calls/service-call-service.ts]
- 정상 근무 귀케어사가 0명인 날의 귀케어 풀 처리는 개인 지급액 0원으로 명시적으로 테스트해야 한다. 이 story에서는 `undistributedAmount`로 남은 귀케어 풀을 표시하고 지급 저장은 하지 않는다. [Source: _bmad-output/project-context.md#테스트-규칙]
- 금액은 VND whole-number `number` 기준이다. 나누어떨어지지 않는 경우 deterministic remainder allocation을 domain service에 둬 표시 금액과 계산 금액이 흔들리지 않게 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story-4.4]
- 정산 화면은 계산 결과와 산출 근거를 항상 함께 보여줘야 하며 맨숫자 금액만 단독으로 보여주면 안 된다. [Source: _bmad-output/planning-artifacts/ux-designs/ux-vietnam_massage-2026-06-07/EXPERIENCE.md#Component-Patterns]
- `settlements` module은 daily therapist, earcare, operations-team calculations를 소유한다. `closing`은 월마감 workflow와 snapshot을 소유하므로 Story 4.4에서 snapshot persistence를 만들면 경계를 위반한다. [Source: _bmad-output/planning-artifacts/architecture.md#Architectural-Boundaries]
- 현재 package baseline은 Next.js `16.2.7`, React `19.2.7`, Prisma `7.8.0`, Zod `4.1.12`, Playwright `1.60.0`, Node test + tsx다. 새 dependency를 추가하지 않는다. [Source: package.json]
- Next.js 공식 Forms guide는 Server Actions 안에서도 인증/인가를 검증해야 한다고 안내한다. 이 story의 새 지급액 service는 read-only지만 기존 근무상태 저장 action은 `requirePermission("payout:write")` 경계를 계속 유지해야 한다. [Source: https://nextjs.org/docs/app/guides/forms]
- 기존 Story 4.3 저장 후 갱신은 `revalidatePath("/settlements/earcare")`를 사용한다. 지급액 결과가 같은 route에 있으므로 근무상태 저장 후 재검증으로 최신 지급액이 반영되어야 한다. [Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
- Prisma transaction은 여러 read/write가 하나의 성공/실패 단위여야 할 때 사용한다. 이 story의 지급액 조회는 read-only라 transaction이 필수는 아니지만, 기존 근무상태 저장+audit transaction은 유지해야 한다. [Source: https://www.prisma.io/docs/orm/prisma-client/queries/transactions]
- Zod `safeParse()`는 성공/실패 result union을 반환한다. 새 service/query validator와 action field error mapping은 기존 패턴처럼 safeParse를 사용한다. [Source: https://zod.dev/basics]
