# Story 2.5: 일별 지출 입력과 요약 계산

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 카운터 또는 관리자,
I want 일별 지출을 입력하고 날짜별 콜/매출/정산 요약을 확인하기를,
so that 원본 `실시간콜입력` U:X 영역의 운영 요약을 ERP에서 같은 기준으로 볼 수 있다.

## Acceptance Criteria

1. **일별 지출 생성**
   - **Given** 카운터 또는 관리자가 콜 원장 화면의 일별 지출 영역에 접근한다
   - **When** 지출 항목을 추가한다
   - **Then** 지출은 일자, 운영월, 지출금액, 내용, 담당자, 비고를 가진다
   - **And** 운영월 날짜 범위를 벗어난 지출은 저장할 수 없다.

2. **금액 저장 형식**
   - **Given** 지출금액이 입력된다
   - **When** 저장한다
   - **Then** 지출금액은 숫자 금액으로 저장된다
   - **And** 표시용 통화 문자열은 UI에서 포맷된다.

3. **지출 수정/비활성과 감사 로그**
   - **Given** 지출 항목이 수정되거나 비활성 처리된다
   - **When** 저장이 완료된다
   - **Then** 일별 지출합계가 갱신된다
   - **And** 행위자, 대상 지출, 변경 전후값, 시각이 감사 로그로 기록된다.

4. **일별 요약**
   - **Given** 선택 날짜에 콜과 지출 데이터가 있다
   - **When** 일별 요약을 조회한다
   - **Then** 예약건수, 방문완료, 노쇼/취소, 결제합계, 마사지사정산, 귀케어풀, 할인합계, 지출합계, 순매출이 표시된다
   - **And** 매출과 정산 관련 수치는 `방문완료` 콜만 기준으로 계산된다.

5. **코스별 요약**
   - **Given** 선택 날짜에 A~E 코스별 방문완료 콜이 있다
   - **When** 코스별 요약을 조회한다
   - **Then** A~E 코스별 방문완료 수, 할인건수, 마사지사 담당 수가 표시된다
   - **And** 비완료 상태의 콜은 코스별 매출/정산 요약에 포함되지 않는다.

6. **순매출 계산**
   - **Given** 지출합계가 존재한다
   - **When** 순매출을 계산한다
   - **Then** 순매출은 방문완료 기준 결제합계에서 지출합계를 반영한 값으로 표시된다
   - **And** 계산 기준은 domain service에 위치한다.

7. **로딩 상태**
   - **Given** 요약 데이터가 로딩 중이다
   - **When** 날짜 또는 운영월이 변경된다
   - **Then** shadcn Skeleton 또는 명확한 로딩 상태가 표시된다
   - **And** 이전 날짜 요약과 새 날짜 요약이 혼동되지 않는다.

8. **검증**
   - **Given** 개발자가 일별 지출과 요약을 검증한다
   - **When** 지출 입력/수정/비활성, 감사 로그, 방문완료 기준 요약, 비완료 제외, 순매출, 코스별 요약 테스트를 실행한다
   - **Then** 모든 테스트가 통과한다
   - **And** 요약 계산은 Excel 행 범위가 아니라 날짜와 운영월 조건으로 수행된다.

### Architecture Guardrails

- `calls` owns service-call ledger, assignments, status history, daily expenses, and completed-call calculations. Do not put this summary in `dashboard`, `settlements`, or UI-only code. [Source: `_bmad-output/planning-artifacts/architecture.md` - Service Boundaries]
- Server Actions are the mutation boundary for daily expense updates. They validate, authorize, call domain services, return `ActionResult<T>`, and revalidate `/calls` after success. [Source: `_bmad-output/planning-artifacts/architecture.md` - API & Communication Patterns]
- Domain services own calculation rules. React components render DTOs and must not recalculate payout/summary totals. [Source: `_bmad-output/planning-artifacts/architecture.md` - Enforcement Guidelines]
- Money values are integer whole VND fields. Formatted display strings are generated in UI, not stored as calculation values. [Source: `_bmad-output/planning-artifacts/architecture.md` - Data Exchange Formats]
- Audit action names use dot notation, audit payloads include actor/target/before/after/timestamp, and audit observes changes but does not calculate domain rules. [Source: `_bmad-output/planning-artifacts/architecture.md` - Audit Event Patterns]
- Current UI uses URL params for selected operating month/date and local state only for transient grid editing. Keep the same pattern for expenses. [Source: `_bmad-output/planning-artifacts/architecture.md` - State Management Patterns]
