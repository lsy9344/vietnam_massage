# Story 1.4: 운영월 관리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want 운영월을 생성하고 상태를 관리하기를,
so that 모든 콜 입력, 정산, 대시보드, 월마감이 같은 운영월 기준으로 계산될 수 있다.

## Acceptance Criteria

1. **운영월 생성**
   - **Given** 관리자가 마스터 설정의 운영월 화면에 접근한다
   - **When** 새 운영월을 생성한다
   - **Then** 운영월은 `YYYY-MM`, 월 시작일, 월 종료일을 가진다
   - **And** 기본 상태는 `작성중`으로 저장된다.

2. **중복 운영월 차단**
   - **Given** 같은 `YYYY-MM` 운영월이 이미 존재한다
   - **When** 관리자가 동일한 운영월을 다시 생성하려 한다
   - **Then** 저장은 차단된다
   - **And** 중복 운영월 오류가 한국어로 표시된다.

3. **운영월 목록 조회**
   - **Given** 운영월 목록이 있다
   - **When** 관리자가 운영월 화면을 조회한다
   - **Then** 운영월, 시작일, 종료일, 현재 상태, 생성/수정 시각이 표시된다
   - **And** 최신 운영월 또는 현재 날짜가 속한 운영월을 쉽게 선택할 수 있다.

4. **작성중에서 검토중 상태 변경**
   - **Given** 운영월 상태가 `작성중`이다
   - **When** 관리자가 상태를 `검토중`으로 변경한다
   - **Then** 상태 변경이 저장된다
   - **And** 이후 콜/정산 화면은 변경된 운영월 상태를 조회할 수 있다.

5. **월마감 상태값 재사용성**
   - **Given** 운영월 상태가 `마감확정` 또는 `잠금`이다
   - **When** 일반 사용자가 지급 영향 데이터 변경을 시도한다
   - **Then** 이후 story에서 구현될 데이터 변경 흐름이 이 상태를 권한/잠금 기준으로 사용할 수 있도록 상태값이 명확히 제공된다
   - **And** 이 story에서는 월마감 계산이나 재오픈 자체를 구현하지 않는다.

6. **감사 로그 기록**
   - **Given** 운영월이 생성되거나 상태가 변경된다
   - **When** 저장이 완료된다
   - **Then** 행위자, 대상 운영월, 변경 전후 상태, 시각이 감사 로그로 기록된다
   - **And** 감사 로그는 일반 운영 경로에서 삭제할 수 없다.

7. **운영월 날짜 범위 제공**
   - **Given** 운영월 날짜 조건이 필요하다
   - **When** 도메인 서비스가 운영월을 조회한다
   - **Then** 월간 집계가 Excel 행 범위가 아니라 운영월 시작일/종료일로 계산될 수 있도록 날짜 범위를 반환한다
   - **And** 날짜는 ISO `YYYY-MM-DD` 형식을 사용한다.

8. **검증**
   - **Given** 개발자가 운영월 기능을 검증한다
   - **When** 생성, 중복 생성 차단, 상태 변경, 감사 로그 기록 테스트를 실행한다
   - **Then** 모든 테스트가 통과한다
   - **And** 운영월 상태값은 월마감 story에서 재사용할 수 있다.

### Epic 1 Context

Epic 1 establishes ERP 운영 기준과 접근 통제 before room, code, employee, course, rate, and incentive masters. Story 1.4 is the first master-data story after audit; it must create the operating-month source of truth used by calls, rooms, settlements, closing, and dashboards. [Source: `_bmad-output/planning-artifacts/epics.md` - Epic 1, Story 1.4]

FR4 requires administrators to create operating months by `YYYY-MM`, month start/end dates, and status values `작성중`, `검토중`, `마감확정`, `잠금`; monthly aggregates must use operating-month date conditions. [Source: `_bmad-output/planning-artifacts/prds/prd-vietnam_massage-2026-06-07/prd.md` - FR-4]

This story must not implement monthly close calculation, confirmation snapshot, lock enforcement across calls/settlements, or admin reopen. ### Architecture Guardrails

- Keep route composition in `src/app/(erp)/masters/operating-months`, domain logic in `src/modules/masters`, cross-cutting authorization in `src/lib`, and immutable change history in `src/modules/audit`. Do not put date-range calculation or status transition rules in React components. [Source: `_bmad-output/planning-artifacts/architecture.md` - Project Organization, Service Boundaries]
- Prisma model names use PascalCase singular and DB tables use snake_case plural with `@@map`. For this story use `OperatingMonth` mapped to `operating_months`. [Source: `_bmad-output/planning-artifacts/architecture.md` - Naming Patterns]
- Data exchange uses camelCase field names, service dates use ISO `YYYY-MM-DD`, and persisted state comes from server/domain services. Selected operating month/date is stored in URL params in later screens, not a global client store. [Source: `_bmad-output/planning-artifacts/architecture.md` - Data Exchange Formats, State Management Patterns]
- Server Actions and Route Handlers must authenticate, authorize, validate, call domain services, and map user-safe Korean errors. Treat Server Actions as public-facing server entrypoints. [Source: `_bmad-output/planning-artifacts/architecture.md` - API & Communication Patterns; External: Next.js Mutating Data - `https://nextjs.org/docs/app/getting-started/mutating-data`]
- Zod 4 is the project baseline for server input validation. Use `safeParse()` or equivalent error mapping for untrusted form data and `z.enum([...])`/regex for status and `YYYY-MM`. [External: Zod 4 release/API - `https://zod.dev/v4`, `https://zod.dev/api`]
- Prisma docs require `DateTime` values to be passed as `Date` objects and support PostgreSQL native `@db.Date` for date-only storage. Convert service DTO dates back to ISO `YYYY-MM-DD`. [External: Prisma Schema API - `https://docs.prisma.io/docs/orm/reference/prisma-schema-reference`]

### Anti-Patterns to Avoid

- Do not use Excel row numbers, sheet coordinates, or generated date rows as the operating-month truth. Use `monthKey`, `startDate`, and `endDate`.
- Do not use display labels as stable keys. `monthKey` and DB `id` are the stable identifiers.
- Do not implement monthly close confirmation, payout snapshot creation, lock enforcement across every module, or reopen in this story.
- Do not allow arbitrary status strings. Only `작성중`, `검토중`, `마감확정`, `잠금` are valid business statuses.
- Do not rely on sidebar hiding for security. Actions and pages must check server-side route/permission access.
- Do not duplicate audit storage or create audit update/delete paths.
- Do not return raw Prisma unique constraint, validation, or infrastructure errors to users.

### Testing Standards

- Minimum static validation: `npm run lint` must include Story 1.1 + 1.2 + 1.3 + 1.4 checks.
- Domain/service tests or static validator must cover:
  - `2026-06` creates start `2026-06-01` and end `2026-06-30`
  - leap-year month `2028-02` creates end `2028-02-29`
  - invalid `YYYY-MM` is rejected with Korean error mapping
  - duplicate month is rejected
  - `작성중 -> 검토중` is accepted
